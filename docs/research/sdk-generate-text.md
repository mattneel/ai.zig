# sdk-generate-text

## Summary
Deep-read of packages/ai/src/generate-text (~69k LOC incl. tests). generateText is a do/while step loop: standardize prompt, replay/validate tool approvals into an initial tool message, per step run prepareStep overrides -> prepareTools/ToolChoice -> retry(doGenerate) -> parseToolCall (with repair + refine, never-throw invalid fallback) -> approval resolution -> parallel client tool execution -> asContent interleaving -> toResponseMessages feeding the next step; continues while all client tool calls have outputs/denials (or deferred provider results pending) and no stopWhen condition (OR-ed array over steps) is met; totalUsage = fieldwise sum; output parsed only on finishReason==='stop'. streamText is a 5-layer pull pipeline built from WHATWG streams: streamLanguageModelCall standardizes provider chunks (LanguageModelV4StreamPart -> LanguageModelStreamPart incl. model-call-start/-response-metadata/-end with performance stats), invokeToolCallbacksFromStream fires tool input callbacks, executeToolsFromStream forwards chunks and executes tools after model-call-end (streaming preliminary results), a per-step transform emits start-step/finish-step and decides continuation (recursive streamStep appended to a stitchable FIFO stream, synchronized with the event processor via a stepFinish DelayedPromise), and the base pipeline adds start/abort handling, stopStream gate, user transforms (smoothStream), output-partial parsing (EnrichedStreamPart), and an event processor that assembles StepResults from deltas and resolves result promises in flush. Result promises are lazy DelayedPromises that auto-consume the stream; derived streams (textStream/stream/partialOutputStream/elementStream) come from repeated tee(). Errors, tool errors, invalid tool calls, and aborts are in-stream data parts, not exceptions; abort emits an abort part and skips onError/onEnd. Captured full chunk vocabularies, StepResult/result type shapes, Output spec interface (text/object/array/choice/json), smoothStream chunking algorithm, timeout/abort merging, and test-verified contracts.

## Key findings
### generateText control flow (single + multi-step loop)
generateText<TOOLS,RUNTIME_CONTEXT,OUTPUT>(options) -> Promise<GenerateTextResult>. Flow: resolveLanguageModel -> asArray(stopWhen) [default isStepCount(1)] -> mergeAbortSignals(abortSignal, totalTimeoutMs, stepAbortController.signal) -> prepareRetries -> standardizePrompt -> collectToolApprovals from incoming messages + validateApprovedToolApprovals; approved ones execute BEFORE step 0 and produce an initial `role:'tool'` response message (denied ones produce output {type:'execution-denied', reason}). Then a do/while loop: each iteration = one step. Per step: setAbortTimeout(step) -> prepareStep?.() (may override model/toolChoice/activeTools/toolOrder/instructions/messages/toolsContext/runtimeContext/sandbox/providerOptions; instructions+messages overrides CARRY FORWARD to later steps) -> convertToLanguageModelPrompt -> filterActiveTools -> prepareTools -> prepareToolChoice -> onStepStart -> onLanguageModelCallStart -> retry(model.doGenerate({...callSettings, tools, toolChoice, responseFormat: await output?.responseFormat, prompt, providerOptions, abortSignal, headers})) -> parse tool calls via parseToolCall (Promise.all over content parts of type 'tool-call') -> onLanguageModelCallEnd -> per tool call: tool.onInputAvailable?, resolveToolApproval (statuses: 'not-applicable'|'user-approval'|'approved'|'denied'; only non-not-applicable consume a generated approvalId, HMAC-signed via maybeSignApproval) -> insert tool-error outputs for invalid+dynamic calls -> executeTools (Promise.all executeToolCall over client (non-providerExecuted), valid, non-blocked calls) -> build stepContent via asContent() (interleaves provider content with parsed calls, client outputs, approval requests/responses; outputs with approval responses ordered AFTER them) -> toResponseMessages -> push DefaultStepResult -> messagesForNextStep = [...stepMessages, ...stepResponseMessages] -> onStepEnd. Loop condition: ((clientToolCalls.length>0 && clientToolOutputs.length+deniedApprovals.length === clientToolCalls.length) || pendingDeferredToolCalls.size>0) && !(await isStopConditionMet({stopConditions, steps})). After loop: totalUsage = reduce addLanguageModelUsage over steps; onEnd with flatMapped aggregates; output parsed ONLY if lastStep.finishReason === 'stop' via (output ?? text()).parseCompleteOutput({text}, {response,usage,finishReason}); result.output getter throws NoOutputGeneratedError if not parsed. Errors wrapped via wrapGatewayError.

### StopCondition / stopWhen
type StopCondition<TOOLS,CTX> = (options: {steps: StepResult[]}) => boolean | PromiseLike<boolean>. Built-ins: isStepCount(n) => steps.length === n; isLoopFinished() => false (run until natural termination); hasToolCall(...names) => last step's toolCalls contains any name. isStopConditionMet evaluates ALL conditions in parallel (Promise.all) and ORs them. Natural termination (independent of stopWhen): finishReason !== 'tool-calls' effectively (loop only continues when every client tool call got an output or denial — a tool without execute produces no output so the loop halts), or a tool call awaiting user approval (blockedToolCallIds keeps output count < call count).

### PrepareStepFunction
prepareStep(options: {steps, stepNumber, model, instructions, initialInstructions, messages, initialMessages, responseMessages, toolsContext, runtimeContext, experimental_sandbox}) -> PrepareStepResult | undefined (sync or Promise). PrepareStepResult fields (all optional; undefined = inherit outer): model, toolChoice, activeTools, toolOrder, instructions (deprecated alias system), messages, toolsContext, runtimeContext, experimental_sandbox (this-step-only), providerOptions (merged with outer via mergeObjects). runtimeContext/toolsContext/instructions/messages mutations persist to subsequent steps.

### Provider stream chunk vocabulary (LanguageModelV4StreamPart)
Raw provider doStream() emits: text-start/text-delta/text-end {id, delta, providerMetadata?}; reasoning-start/-delta/-end (same shape); tool-input-start {id, toolName, providerExecuted?, dynamic?, title?}, tool-input-delta {id, delta}, tool-input-end {id}; tool-call (LanguageModelV4ToolCall: toolCallId, toolName, input: string(JSON), providerExecuted?, dynamic?); tool-result (provider-executed: toolCallId, toolName, result, isError?, dynamic?); tool-approval-request; custom; file/reasoning-file {data: {type:'data',data}|{url}, mediaType}; source; stream-start {warnings}; response-metadata {id?, timestamp?, modelId?}; finish {usage: LanguageModelV4Usage, finishReason: {unified, raw}, providerMetadata?}; raw {rawValue}; error {error}. Concurrent text/reasoning/tool-input blocks are correlated by string `id`; a tool-input block's id == the eventual toolCallId.

### streamText 5-layer pipeline architecture
Layer 1 streamLanguageModelCall: standardizePrompt + convertToLanguageModelPrompt + prepareTools/ToolChoice, calls model.doStream(), returns {stream: AsyncIterableStream<LanguageModelStreamPart>, request:{body?}, response:{headers?}}; pipes raw parts through a TransformStream that: renames delta->text on text/reasoning deltas, wraps files in DefaultGeneratedFileWithType, parses tool-call parts via parseToolCall (invalid ones emit the tool-call THEN a synthetic tool-error part), tracks toolCallsByToolCallId so provider tool-results/approval-requests get the parsed input attached, accumulates modelCallContent for onLanguageModelCallEnd, converts stream-start->model-call-start{warnings}, response-metadata->model-call-response-metadata, finish->model-call-end{finishReason, rawFinishReason, usage: asLanguageModelUsage, providerMetadata, performance{responseTimeMs, effective/outputTokensPerSecond, inputTokensPerSecond, timeToFirstOutputMs, timeBetweenOutputChunksMs stats(min/p10/median/avg/p90/max via nearest-rank)}}. Layer 2 invokeToolCallbacksFromStream: passthrough that fires tool.onInputStart/onInputDelta/onInputAvailable keyed by tool-input id. Layer 3 executeToolsFromStream: passthrough TransformStream that forwards every chunk immediately; on valid client tool-call resolves approval (may emit tool-approval-request/-response parts; denied/user-approval calls are NOT executed) and queues into toolCallsToExecute; on 'model-call-end' runs Promise.all(executeToolCall) for queued calls, enqueuing preliminary tool-results as they stream, then 'tool-execution-end' {toolCallId, toolExecutionMs} + final tool-result/tool-error. Tool results thus appear AFTER model-call-end within the same inner stream; step is only finished when the inner stream flushes. Layer 4 per-step TransformStream in DefaultStreamTextResult.streamStep: swallows model-call-start (captures warnings), emits 'start-step' {request, warnings} before first real chunk, forwards content chunks (drops empty text-deltas; raw only if include.rawChunks), records stepToolCalls/stepToolOutputs (excluding preliminary)/approvalResponses/toolExecutionMs, captures model-call-end into stepUsage/finishReason/providerMetadata/performance; flush(): if no terminal chunk (model-call-end or error) AND no output chunk -> enqueue error NoOutputGeneratedError + closeStream; else enqueue 'finish-step' {finishReason, rawFinishReason, usage, performance{stepTimeMs, toolExecutionMs, ...modelCallPerformance}, providerMetadata, response{id,timestamp,modelId,headers}}; await stepFinish.promise (DelayedPromise resolved by eventProcessor after it built the StepResult — synchronizes continuation decision with step assembly); then decide: continue -> recursive streamStep({currentStep+1, usage: combinedUsage}) adding a NEW inner stream to the stitchable stream; else enqueue 'finish' {finishReason, rawFinishReason, totalUsage} and closeStream(). Layer 5 base pipeline (constructor): stitchableStream -> resilient wrapper ReadableStream (emits {type:'start'} in start(); pull() reads and on abortSignal.aborted or AbortError emits onAbort callback + {type:'abort', reason} then controller.close()) -> stopStream gate (isRunning flag) -> user transforms (experimental_transform, each gets {tools, stopStream(){stitchable.terminate(); isRunning=false}}) -> createOutputTransformStream (wraps every part as EnrichedStreamPart {part, partialOutput}) -> eventProcessor TransformStream (forwards chunk first, then: onChunk callback for EVERY part; on 'error' part wrap + onError callback; accumulates recordedContent — text/reasoning assembled from start/delta/end by id into mutable {type,text,providerMetadata} objects, missing id => synthetic error part 'text part <id> not found'; on 'start-step' resets per-step recorded state; on 'finish-step' builds DefaultStepResult, calls onStepFinish, pushes to recordedSteps, computes stepMessagesForNextStep, resolves stepFinish; on 'finish' records totals) with flush() that resolves the result DelayedPromises (steps/finishReason/rawFinishReason/totalUsage) or rejects them all (abort reason, recordedNoOutputError, or generic NoOutputGeneratedError) when recordedSteps empty, then fires onEnd with flatMapped aggregates.

### Stitchable stream (multi-step splice)
createStitchableStream<T>() -> {stream: ReadableStream<T>, addStream(inner), close(), terminate()}. Maintains FIFO innerStreamReaders[]; outer pull() reads from head reader; when head is done, shifts and recurses; when empty and not closed, awaits a resolvable promise signaled by addStream/close. close() = graceful (drain remaining inner streams then close); terminate() = cancel all inner readers and close outer immediately. Inner-stream error -> controller.error + terminate. This is the mechanism by which step N+1's stream is appended while step N's flush is still executing; pull-based lazy evaluation gives end-to-end backpressure (nothing is read from the provider stream until the consumer pulls; onChunk pauses processing since transform() awaits it).

### Result promise semantics / lazy consumption
DefaultStreamTextResult holds DelayedPromise fields (_totalUsage, _finishReason, _rawFinishReason, _steps, _initialResponseMessages). DelayedPromise only materializes the underlying Promise when .promise is accessed (avoids unhandled rejections). Accessor getters steps/totalUsage/finishReason/rawFinishReason call this.consumeStream() (fire-and-forget full drain with onError -> rejectResultPromises) so awaiting any promise auto-drains the stream. Derived promises: finalStep=steps.at(-1), text/reasoningText/request/response/providerMetadata from finalStep; content/files/sources/toolCalls/toolResults/warnings = flatMap over steps; responseMessages = initialResponseMessages ++ steps.flatMap(s=>s.response.messages); usage aliases totalUsage. Streams are derived by teeing: teeStream() does [s1,s2]=baseStream.tee(); baseStream=s2; return s1 — every access to .stream/.textStream/.partialOutputStream/.elementStream/.toUIMessageStream gets an independent copy (buffers server-side; documented as acceptable for LLM-sized data). stream/fullStream = tee -> map EnrichedStreamPart->part; textStream = toTextStream(stream) = filter type==='text-delta' -> part.text; partialOutputStream = tee -> filter partialOutput!=null; elementStream = tee -> output.createElementStreamTransform() else throw UnsupportedFunctionalityError. All are AsyncIterableStream<T> = ReadableStream<T> + Symbol.asyncIterator (iterator cancels stream on early break/throw).

### TextStreamPart union (public fullStream vocabulary)
TextStreamPart<TOOLS> = text-start|text-end|text-delta{id,text,providerMetadata?} | reasoning-start|-end|-delta | custom{kind:'x.y'} | tool-input-start{id,toolName,toolMetadata?,providerExecuted?,dynamic?,title?} | tool-input-end{id} | tool-input-delta{id,delta} | source | file{file:GeneratedFile} | reasoning-file | tool-call(TypedToolCall) | tool-result(TypedToolResult, has preliminary?) | tool-error(TypedToolError) | tool-output-denied | tool-approval-request{approvalId,toolCall,isAutomatic?,signature?} | tool-approval-response{approvalId,approved,reason?,providerExecuted?} | start-step{request,warnings} | finish-step{response,usage,performance,finishReason,rawFinishReason,providerMetadata} | start{} | finish{finishReason,rawFinishReason,totalUsage} | abort{reason?} | error{error} | raw{rawValue}. Canonical order per test: start, start-step, (content parts...), finish-step, [next step: start-step...finish-step], finish. onChunk receives every part including start/start-step/finish-step/finish.

### Tool call parsing/repair/refinement
parseToolCall({toolCall: LanguageModelV4ToolCall, tools, repairToolCall?, refineToolInput?, instructions, messages}) -> TypedToolCall. doParseToolCall: unknown toolName + providerExecuted+dynamic => parse as dynamic provider tool; unknown otherwise => NoSuchToolError. input string '' -> validate {} against schema (LLMs emit empty strings for no-arg tools); else safeParseJSON with schema -> InvalidToolInputError on failure. On NoSuchToolError/InvalidToolInputError and repairToolCall provided: repairToolCall({toolCall, tools, inputSchema: async fn, instructions, system, messages, error}) -> repaired LanguageModelV4ToolCall | null (null => rethrow; repair throw => ToolCallRepairError{cause, originalError}). Final catch: NEVER throws — returns {type:'tool-call', dynamic:true, invalid:true, error, input: best-effort-parsed} so invalid calls flow through the stream as data. refineToolInput: per-tool-name map input=>input applied after successful parse. TypedToolCall = StaticToolCall (toolName is keyof TOOLS, input typed) | DynamicToolCall {input: unknown, dynamic:true, invalid?, error?}.

### executeToolCall lifecycle
executeToolCall({toolCall, tools, callId, messages, abortSignal, timeout, sandbox, toolsContext, onPreliminaryToolResult?, onToolExecutionStart?, onToolExecutionEnd?, ...telemetry}) -> Promise<{output: ToolOutput, toolExecutionMs} | undefined>. Returns undefined when tool has no execute (isExecutableTool false) — this stalls the loop by design. validateToolContext against tool.contextSchema. Merges abortSignal with per-tool timeout getToolTimeoutMs(timeout, toolName) (timeout.tools['<name>Ms'] ?? timeout.toolMs). Executes via provider-utils executeTool async generator: if tool.execute returns AsyncIterable, each yield => {type:'preliminary', output} (forwarded as tool-result{preliminary:true} parts) and last value re-emitted as final; else single final. On throw: returns {output: {type:'tool-error', toolCallId, toolName, input, error, dynamic}} (tool errors are DATA, not exceptions). notify() runs all callbacks in parallel and swallows their errors. ToolOutput<TOOLS> = TypedToolResult | TypedToolError.

### Tool choice / active tools / tool order
toolChoice?: ToolChoice<TOOLS> = 'auto'(default)|'none'|'required'|{type:'tool', toolName}. prepareToolChoice normalizes before provider call. activeTools: subset of tool names sent to provider without changing result types (filterActiveTools). toolOrder: partial ordering list; unlisted tools appended alphabetically (provider cache stability). All three overridable per step via prepareStep.

### Output specification (experimental_output)
interface Output<COMPLETE, PARTIAL, ELEMENT> {name: string; responseFormat: PromiseLike<{type:'text'} | {type:'json', schema?, name?, description?}>; parseCompleteOutput({text}, {response, usage, finishReason}) -> Promise<COMPLETE> (throws NoObjectGeneratedError on parse/validation failure); parsePartialOutput({text}) -> Promise<{partial: PARTIAL} | undefined>; createElementStreamTransform() -> TransformStream<EnrichedStreamPart, ELEMENT> | undefined}. Implementations: text() (identity; partial=accumulated text), object({schema}) (partial via parsePartialJson with states failed-parse|undefined-input|repaired-parse|successful-parse; no partial validation), array({element}) (wraps schema as {elements: [...]}; partial validates each element, drops last element when state==='repaired-parse'; elementStream transform emits only new completed elements by index), choice({options}) (schema {result: enum}; partial matched by prefix, repaired parse only if unambiguous), json(). Streaming hook: createOutputTransformStream buffers text-deltas of the FIRST text part id only (other text ids pass through), calls parsePartialOutput on the accumulated text after each delta, and re-emits the buffered textChunk as a single text-delta only when partial output changed (comparison via string identity or JSON.stringify), pairing it with partialOutput; buffered text force-flushed on text-end and before finish-step. responseFormat is passed to doGenerate/doStream. In generateText, output parsed only when finishReason==='stop'; streamText result.output = finalStep-based parseCompleteOutput (no finish-reason gate).

### smoothStream
smoothStream({delayInMs=10 (null=no delay), chunking='word'|'line'|RegExp|ChunkDetector|Intl.Segmenter}) -> (options:{tools}) => TransformStream<TextStreamPart, TextStreamPart>. Used as experimental_transform. Buffers only text-delta and reasoning-delta; any other chunk type flushes buffer then passes through; buffer also flushed when delta type or id changes. detectChunk('word')=/\S+\s+/m match => emit buffer.slice(0,match.index)+match[0]; loop emitting detected chunks with await delay(delayInMs) between each; trailing partial word stays buffered until a non-delta chunk (e.g. text-end) flushes it. providerMetadata preserved on flushed chunk (Anthropic thinking signatures). ChunkDetector must return prefix of buffer, non-empty (else throw). Transforms run BEFORE output parsing so replaced segments feed partial-output parsing.

### Abort / timeout handling
TimeoutConfiguration = number (totalMs) | {totalMs?, stepMs?, chunkMs?, toolMs?, tools?: {'<name>Ms': number}}. mergeAbortSignals(...(AbortSignal|number|null)[]) -> AbortSignal.any of signals + AbortSignal.timeout(n) conversions. streamText merges caller signal + totalTimeout + stepAbortController + chunkAbortController. Step timer armed per streamStep, cleared in flush before next step; chunk timer reset on EVERY chunk in the per-step transform (resetChunkTimeout) — chunkMs = max gap between chunks. Timers also cleared via abortSignal 'abort' listeners since step streams outlive streamStep() setup. Abort contract (from tests): abort does NOT call onError; emits {type:'abort', reason: getErrorMessage(signal.reason)} part then closes stream; onAbort({steps: recordedSteps, reason?}) fired; result promises reject with abortSignal.reason when no steps recorded. isAbortError distinguishes AbortError from real errors in the resilient wrapper's pull().

### Error propagation contract
Two channels: (1) in-stream {type:'error', error} parts — provider stream errors, doStream throw (caught by the outer async IIFE .catch which appends a one-part error stream and closes), tool execution errors during recursion, invalid tool calls; these do NOT throw from fullStream iteration; onError callback invoked per error part (default onError logs to console.error); step continues to finish-step with finishReason:'error'. (2) Thrown errors — only stream-machinery failures propagate through controller.error and reject result promises. textStream throws on error (no error parts representation). Incomplete model stream (no model-call-end/error terminal chunk): if NO output chunks received => synthetic NoOutputGeneratedError error part + close (no step recorded); if partial output exists => step retained. onFinish/onEnd still fires after mid-stream error parts (test: 'should call onFinish even when error chunk occurs mid-stream'). eventProcessor.flush rejects all result promises when recordedSteps.length===0 or recordedNoOutputError set.

### StepResult shape
StepResult<TOOLS,CTX> = {callId, stepNumber, model:{provider,modelId}, toolsContext, runtimeContext, content: ContentPart[], text (derived: join of text parts), reasoning (derived), reasoningText, files, sources, toolCalls, staticToolCalls (dynamic!==true), dynamicToolCalls, toolResults, staticToolResults, dynamicToolResults, finishReason: FinishReason, rawFinishReason: string|undefined, usage: LanguageModelUsage, performance: StepResultPerformance, warnings, request: LanguageModelRequestMetadata, response: LanguageModelResponseMetadata (incl. messages), providerMetadata}. DefaultStepResult stores content + metadata; all list accessors are getters that filter content by part.type — content is the single source of truth. ContentPart union = text|custom|reasoning|reasoning-file|source|file|tool-call|tool-result|tool-error|tool-approval-request|tool-approval-response.

### Usage accumulation
LanguageModelUsage = {inputTokens?, inputTokenDetails:{noCacheTokens?, cacheReadTokens?, cacheWriteTokens?}, outputTokens?, outputTokenDetails:{textTokens?, reasoningTokens?}, totalTokens?, raw?: JSONObject}. All fields number|undefined. addLanguageModelUsage: fieldwise addTokenCounts where undefined+undefined=undefined, else (a??0)+(b??0); raw is dropped when summing. asLanguageModelUsage converts provider LanguageModelV4Usage {inputTokens:{total,noCache,cacheRead,cacheWrite}, outputTokens:{total,text,reasoning}}. streamText threads accumulated usage through streamStep({currentStep, usage}) recursion; combinedUsage = addLanguageModelUsage(usage, stepUsage) becomes finish.totalUsage; generateText reduces over steps.

### toResponseMessages (feedback into next-step prompt)
toResponseMessages({content, tools}) -> [AssistantModelMessage?, ToolModelMessage?]. Assistant message: text (skip empty), custom, reasoning, file/reasoning-file (as base64), tool-call (invalid non-object input replaced with {}), provider-executed tool-result/tool-error (as tool-result with output via createToolModelOutput errorMode 'none'/'json'), tool-approval-request {approvalId, toolCallId, isAutomatic?, signature?}. Tool message: client (non-providerExecuted) tool-result/tool-error (errorMode 'text' for errors) + tool-approval-response entries; denied approvals additionally append tool-result {output:{type:'execution-denied', reason}}. Tool results sorted by original tool-call order (toolCallOrder map, stable for unknown ids). Response messages deep-cloned (cloneModelMessages) before storage in step results to avoid mutation across steps.

### generateText continuation nuances (deferred provider tools)
pendingDeferredToolCalls: Map<toolCallId,{toolName}> tracks provider-executed calls on tools with tool.type==='provider' && tool.supportsDeferredResults whose result did not arrive in the same response (programmatic tool calling, e.g. code_execution triggering a client tool); loop/stream continues while non-empty even without client tool calls; entries removed when a tool-result for that id arrives in a later step. In asContent, a tool-result with no matching tool-call is only legal for such tools (input: undefined), else throws 'Tool call <id> not found.'.

### GenerateTextResult / StreamTextResult surfaces
GenerateTextResult (sync interface, getters over steps): content, text (finalStep), reasoning/reasoningText (finalStep), files/sources/toolCalls/staticToolCalls/dynamicToolCalls/toolResults/static/dynamic (flatMap all steps), finishReason/rawFinishReason (finalStep), usage===totalUsage (sum), warnings (flatMap), request/response/providerMetadata (finalStep), responseMessages (initial + all steps), steps, finalStep, output (throws NoOutputGeneratedError if absent). StreamTextResult: identical fields but each is PromiseLike + streams: textStream, stream (fullStream deprecated alias), partialOutputStream, elementStream, output: PromiseLike, consumeStream(options?), plus HTTP helpers toUIMessageStream/toUIMessageStreamResponse/pipeUIMessageStreamToResponse/toTextStreamResponse/pipeTextStreamToResponse (all deprecated in favor of standalone helpers over result.stream).

### Callback set and ordering
Callbacks (both functions, with deprecated experimental_ aliases): onStart (before any LLM call), onStepStart (per step, receives promptMessages/stepTools/stepToolChoice), onLanguageModelCallStart/End (per provider call; End receives normalized content+usage+performance before client tool execution), onToolExecutionStart/End (End is discriminated by toolOutput type tool-result vs tool-error, includes toolExecutionMs), onStepEnd/onStepFinish (receives StepResult), onEnd/onFinish (final aggregate event incl. steps, totalUsage, responseMessages, finalStep), streamText-only: onChunk (every TextStreamPart; awaited => pauses pipeline), onError (error parts), onAbort ({steps, reason?}). All dispatched via notify() = parallel Promise.all with per-callback try/catch (callback errors never break generation). Telemetry dispatcher mirrors each callback and wraps model/tool calls in spans (runInTracingChannelSpan / executeLanguageModelCall / executeTool contexts).

### Test-verified behavioral contracts (selection)
(1) doStream throw => fullStream yields [start, error{...}] and textStream is empty, never throws to server (onError swallows). (2) Multiple stream consumption: textStream, stream, toUIMessageStream all usable on one result (tee). (3) Missing text/reasoning part id in delta => in-stream error part `text part <id> not found` (createIdMap = null-prototype map to avoid Object.prototype pollution). (4) finish-step carries performance with timeToFirstOutputMs and timeBetweenOutputChunksMs stats. (5) Abort mid-stream: fullStream = [start, start-step, abort{reason}]; onAbort called with steps=[]; telemetry onAbort fires but NOT onEnd/onError. (6) empty text deltas are dropped from fullStream. (7) onChunk sees start-step before content, tool-input parts interleaved with reasoning by id. (8) usage snapshot: totalTokens = input+output; finish.totalUsage sums steps. (9) tool errors surface as tool-error parts and finishReason stays 'tool-calls' (loop can continue with error output fed back as tool message errorMode 'text').

## Port implications
The whole design reduces to three primitives Zig must supply: (1) a typed chunk union, (2) composable pull-based stream stages, (3) promise-like one-shot result cells. TextStreamPart/LanguageModelV4StreamPart/ContentPart map cleanly to Zig tagged unions (zig-best-practices style); make the union exhaustive so new chunk types force classification (the SDK does this deliberately with `isOutputChunkType satisfies Record<...>`). The WHATWG TransformStream chain (raw->standardize->tool-callbacks->tool-exec->step-assembly->gate->transforms->output-parse->event-processor) should become a chain of pull-driven stage structs over std.Io: each stage owns a `next(io) !?Part` (or Reader-style vtable) pulling from its upstream; backpressure then falls out naturally, exactly like the JS pull() semantics. The stitchable stream is a FIFO of child streams plus a condition ('stream added or closed'); in std.Io model this is a mutex+condition or channel where streamStep recursion becomes an explicit loop that appends the next step's stage (avoid actual recursion; the JS recursion is just continuation scheduling). DelayedPromise maps to a one-shot future (mutex+cond+?value) with explicit reject-with-error; 'accessing a promise auto-consumes the stream' becomes an explicit `drain()` in getter methods. tee() is the hardest mismatch: JS tee buffers unboundedly; in Zig prefer either an allocator-backed broadcast buffer with documented ownership, or restrict to single consumption per derived stream and record parts for the result object (the eventProcessor already records everything needed for StepResult — you can build text/steps promises from recordedContent instead of teeing). AbortSignal.any + timeouts map to a cancellation token checked at each pull plus std.Io timer/deadline support; chunk timeout = deadline reset per pull. Errors-as-data is a gift for Zig: in-stream `error` parts are values, tool errors are values, and only machinery failures use Zig `error`; parseToolCall's never-throw fallback (invalid:true tool-call) fits tagged unions perfectly. Tool execution concurrency (Promise.all after model-call-end, preliminary results streamed) needs an Io-based task group whose results are enqueued into the current stage — a bounded queue between tool tasks and the step stream. Dynamic typing pain points: TOOLS generic maps to comptime toolset struct (static typing of input/output per tool) plus a dynamic JSON-value variant for dynamic/invalid tools; Output<COMPLETE,PARTIAL,ELEMENT> maps to comptime interface with parsePartialJson requiring an incremental/repairing JSON parser (port parse-partial-json — it is load-bearing for partialOutputStream/elementStream). For C FFI, the fullStream vocabulary is the natural ABI: expose a `ai_stream_next()` returning a C-repr tagged struct per TextStreamPart, plus callback registration mirroring onChunk/onStepFinish/onFinish; Python/Rust wrappers then rebuild async iterators over that pull API. Callbacks must be documented as pipeline-pausing (the JS awaits them inline). Deep-clone of response messages between steps implies arena-per-step allocation with explicit copies into a call-lifetime arena for steps/results.
