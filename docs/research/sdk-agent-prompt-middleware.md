# sdk-agent-prompt-middleware

## Summary
Deep-read of packages/ai/src {agent, prompt, middleware, registry, error, types} plus the provider-utils message/content types and @ai-sdk/provider error/middleware bases they re-export. The Agent surface is an interface (version 'agent-v1', id, tools, generate(), stream()) with ToolLoopAgent as the canonical implementation: it stores ToolLoopAgentSettings, validates call options against an optional schema, runs an optional prepareCall hook to rewrite call args, merges settings-level and call-level lifecycle callbacks (onStart/onStepStart/onToolExecutionStart/End/onStepEnd/onEnd, each with deprecated aliases), injects a 'ai-sdk-agent/tool-loop' user-agent suffix header, defaults stopWhen to isStepCount(20), and delegates to generateText/streamText. The prompt layer defines ModelMessage (system|user|assistant|tool) with content-part unions, standardizePrompt (prompt-vs-messages exclusivity, system-message policing, zod validation), and convertToLanguageModelPrompt which downloads URL assets, maps approval ids, merges consecutive tool messages, enforces tool-result completeness (MissingToolResultsError), and lowers everything (incl. ~9 deprecated tool-output content types) to the LanguageModelV4Prompt shape. Middleware is a struct of optional hooks (transformParams/wrapGenerate/wrapStream + overrideProvider/ModelId/SupportedUrls); wrapLanguageModel folds an array right-to-left around the model; built-ins include defaultSettingsMiddleware, extractReasoningMiddleware (incremental tag-scanning stream state machine), simulateStreamingMiddleware, extractJsonMiddleware (fence-stripping with 12-char suffix buffer), addToolInputExamplesMiddleware. Registry splits 'providerId:modelId' at the first separator and optionally wraps returned models in middleware; customProvider does map lookup with fallback provider; string model ids resolve via globalThis.AI_SDK_DEFAULT_PROVIDER ?? gateway. All errors extend AISDKError using Symbol.for('vercel.ai.error.AI_<Name>') markers; full field inventory captured.

## Key findings
### Agent interface (agent/agent.ts)
interface Agent<CALL_OPTIONS=never, TOOLS extends ToolSet={}, RUNTIME_CONTEXT extends Context=Context, OUTPUT extends Output=never> { readonly version: 'agent-v1'; readonly id: string|undefined; readonly tools: TOOLS; generate(AgentCallParameters) -> PromiseLike<GenerateTextResult<TOOLS,RUNTIME_CONTEXT,OUTPUT>>; stream(AgentStreamParameters) -> PromiseLike<StreamTextResult<...>> }. AgentCallParameters = ({options} required iff CALL_OPTIONS != never) & exactly-one-of {prompt: string|ModelMessage[]} | {messages: ModelMessage[]} & { abortSignal?, timeout?: TimeoutConfiguration<TOOLS>, onStart?, onStepStart?, onToolExecutionStart?, onToolExecutionEnd?, onStepEnd?, onEnd?, experimental_sandbox? } (each callback has a deprecated experimental_/onFinish/onStepFinish alias). AgentStreamParameters adds experimental_transform?: Arrayable<StreamTextTransform<TOOLS>>.

### ToolLoopAgent control flow (agent/tool-loop-agent.ts)
Constructor: settings copied with onEnd defaulted from onFinish. prepareCall(options): (1) if settings.callOptionsSchema != null and options.options !== undefined -> validateTypes(options.options); (2) strip all lifecycle callbacks from settings; (3) baseCallArgs = settingsWithoutCallbacks + stopWhen ?? isStepCount(20) + call options; (4) preparedCallArgs = await settings.prepareCall?.(baseCallArgs) ?? baseCallArgs; (5) split out {instructions, allowSystemInMessages, messages, prompt, runtimeContext}; runtimeContext key omitted entirely when undefined. generate()/stream(): callbacks merged via mergeCallbacks(settingsCb ?? deprecatedAlias, callCb ?? deprecatedAlias) — mergeCallbacks runs all in parallel with Promise.allSettled, errors swallowed. headers pass through withUserAgentSuffix(headers, 'ai-sdk-agent/tool-loop'). Loop termination doc: finish reason != tool-calls, tool without execute, tool needs approval, or stopWhen met (default isStepCount(20)).

### Prompt type + standardizePrompt (prompt/prompt.ts, standardize-prompt.ts)

### convertToLanguageModelPrompt control flow (prompt/convert-to-language-model-prompt.ts)
async ({prompt: StandardizedPrompt, supportedUrls: Record<string,RegExp[]>, download: DownloadFunction|undefined, provider?}) -> LanguageModelV4Prompt. Steps: (1) downloadAssets: collect FileParts from user messages, tool-result 'content' outputs (in tool AND assistant messages), convert to tagged data, keep type:'url' ones, compute isUrlSupportedByModel per supportedUrls, call download(plannedDownloads) in parallel -> map url->{mediaType,data:Uint8Array}; (2) build approvalId->toolCallId map from assistant tool-approval-request parts, then approvedToolCallIds set from tool-message tool-approval-response parts; (3) prepend instructions as system message(s); (4) convert each ModelMessage via convertToLanguageModelMessage (string content -> [{type:'text'}]; ImagePart rewritten to FilePart with mediaType ?? 'image' + deprecation warning; empty text parts filtered — assistant keeps empty text if providerOptions set; tool-approval-request parts dropped from assistant; tool-approval-response kept only if providerExecuted); (5) merge consecutive role:'tool' messages into one; (6) track unresolved non-providerExecuted tool-call ids, delete on matching tool-result, subtract approvedToolCallIds, throw MissingToolResultsError if any remain at user/system boundary or at end; (7) drop tool messages left with empty content. convertPartToLanguageModelPart replaces url data with downloaded bytes, sniffs image mediaType via detectMediaType, throws if mediaType missing. mapToolResultOutput lowers all 9 deprecated content variants to {type:'file', data: tagged, mediaType} with deprecation warnings, inferring media type from URL extension table (fallback application/octet-stream).

### File data lowering (prompt/file-part-data.ts, split-data-url.ts, data-content.ts)
convertToLanguageModelV4FilePart(content) -> {data: LanguageModelV4FilePart['data'], mediaType: string|undefined}: tagged shapes pass through (inline 'data' with a 'data:' URL string throws InvalidDataContentError); URL with protocol 'data:' is split into {mediaType, base64Content} (both required or InvalidDataContentError); bare string tried as URL first, else treated as base64 data; ArrayBuffer/Buffer normalized to Uint8Array; bare Record<string,string> detected via isProviderReference -> {type:'reference'}. convertDataContentToUint8Array: string base64-decoded (InvalidDataContentError on failure), ArrayBuffer wrapped, others throw. cloneModelMessages deep-clones preserving URL/Uint8Array/ArrayBuffer/Date.

### LanguageModelV4Middleware + wrapLanguageModel (provider/.../language-model-v4-middleware.ts, ai/middleware/wrap-language-model.ts)
Middleware = { specificationVersion: 'v4'; overrideProvider?({model})->string; overrideModelId?({model})->string; overrideSupportedUrls?({model})->Record<string,RegExp[]> (may be Promise); transformParams?({type:'generate'|'stream', params: LanguageModelV4CallOptions, model})->Promise<CallOptions>; wrapGenerate?({doGenerate, doStream, params, model})->Promise<LanguageModelV4GenerateResult>; wrapStream?(same)->Promise<LanguageModelV4StreamResult> }. ai-package LanguageModelMiddleware relaxes specificationVersion to optional string. wrapLanguageModel({model: V2|V3|V4, middleware: one-or-array, modelId?, providerId?}) -> LanguageModelV4: upconverts via asLanguageModelV4, then array.reverse().reduce(doWrap) so first middleware transforms input first / last is innermost. doWrap returns a new v4 model object: provider = providerId ?? overrideProvider ?? model.provider (same for modelId, supportedUrls); doGenerate/doStream each run transformParams then hand both doGenerate and doStream thunks (bound to transformed params) to the wrap hook, defaulting to the corresponding direct call. wrapEmbeddingModel (transformParams/wrapEmbed + overrideMaxEmbeddingsPerCall/overrideSupportsParallelCalls) and wrapImageModel (transformParams/wrapGenerate + overrideMaxImagesPerCall, function values re-bound to inner model) are analogous. wrapProvider applies language+image middleware to every model of a ProviderV4.

### Built-in middlewares (middleware/*.ts)

## Port implications
Nearly every type here is a closed discriminated union on a string `type`/`role` tag — this maps 1:1 to Zig tagged unions (ModelMessage, content parts, ToolResultOutput, FileData, Warning, stream parts, ToolChoice, FinishReason as enum). Hard parts and design constraints: (1) The error hierarchy carries rich per-error payloads (url, statusCode, toolCallIds[], usage, response metadata); Zig error sets can enumerate the ~35 names, but payloads need a parallel diagnostic struct (e.g. an out-param `*ErrorInfo` union or per-call `error_detail` field on the result object) — the Symbol marker trick becomes irrelevant since Zig has nominal types, but the C FFI needs a stable error-code enum + a getter for the payload union. (2) Middleware is a struct of optional hooks receiving `doGenerate`/`doStream` continuation thunks; in Zig this is naturally a vtable struct with optional function pointers + userdata pointer, and wrapLanguageModel's reverse-reduce becomes building a chain of wrapper model structs (each owning inner pointer) — allocation/ownership must be explicit (arena or caller-owned chain). (3) TS structural spread/merge semantics (prepareCall deleting fields by returning undefined, defaultSettingsMiddleware mergeObjects, callback aliasing/merging) don't port literally; use explicit option structs with optionals and a documented merge function. (4) ProviderOptions/ProviderMetadata are `Record<string, JSONObject>` — needs std.json.Value or StringArrayHashMap(JsonObject); JSONValue recursion is already solved by std.json. (5) DataContent collapses to `[]const u8` (bytes) vs base64 string vs URL vs provider-reference map — a 4-variant union; Buffer/ArrayBuffer distinctions vanish. (6) extractReasoning/extractJson stream middlewares are incremental per-id state machines over stream parts with partial-tag lookahead and a 12-byte suffix holdback — these port cleanly onto std.Io Reader/Writer streaming with per-id hash maps, and are good early tests of the streaming abstraction. (7) convertToLanguageModelPrompt is async because of parallel URL downloads — under std.Io this is the natural place for concurrent fetches (Io.async per planned download). (8) Registry/customProvider are just string-split-at-first-separator + hash-map lookups; the generic template-literal typing has no Zig analogue and should degrade to runtime string ids. (9) The Agent generic parameters (CALL_OPTIONS schema validation, prepareCall) suggest comptime-generic Agent(Tools, Output) in Zig but a type-erased ABI surface for C FFI. (10) TimeoutConfiguration's `${toolName}Ms` keys should become an explicit per-tool StringHashMap(u64). (11) UnsupportedModelVersionError lacks the marker pattern in TS — an upstream inconsistency, harmless in Zig. (12) Usage arithmetic uses `number|undefined` tri-state; use ?u64 with a helper mirroring addTokenCounts (both-null -> null).
