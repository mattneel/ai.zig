# sdk-providers-concrete

## Summary
Both providers implement the same LanguageModelV4 interface (specificationVersion 'v4') and follow a rigid recipe: (1) a factory function (createAnthropic/createOpenAI) closes over settings (baseURL with env-var fallback and trailing-slash strip, apiKey via loadApiKey env fallback, lazy getHeaders() closure, injectable fetch, optional provider `name`) and returns a callable provider object whose methods construct model class instances with a config struct; (2) each model class has getArgs() which collects warnings, parses namespaced providerOptions against a schema, consults a static per-modelId capabilities table, converts the spec prompt to the wire format, prepares tools/toolChoice, and assembles the JSON body; (3) doGenerate POSTs the body via postJsonToApi with a failedResponseHandler (provider error schema -> APICallError) and a JSON success handler, then maps typed response content parts to spec content parts; (4) doStream POSTs with stream flag, gets a ReadableStream<ParseResult<Chunk>> from an SSE handler, and runs a stateful TransformStream that maps provider events to spec stream parts (text-start/delta/end, reasoning-start/delta/end, tool-input-start/delta/end, tool-call, tool-result, source, response-metadata, finish, error, raw), with early-error detection by peeking the first chunks via stream tee(). Anthropic is index-keyed (content_block_start/delta/stop + message_start/delta/stop + ping/error) with a betas Set that accumulates from options/tools/prompt features into the anthropic-beta header; OpenAI Chat is choices[0].delta-based with a StreamingToolCallTracker; OpenAI Responses is item-based (response.output_item.added/done + typed per-field delta events, output_index-keyed state, item_reference payload optimization when store=true). openai-compatible is the vendor template: pluggable errorStructure, metadataExtractor, transformRequestBody, convertUsage, unknown-option passthrough into the body, and reasoning_content handling.

## Key findings
### Provider factory contract
createAnthropic(options)/createOpenAI(options) return a callable-object provider: `const provider = function(modelId){...}` plus properties: specificationVersion:'v4', languageModel, chat, messages/responses/completion, embeddingModel (Anthropic throws NoSuchModelError), imageModel, files(), skills(), tools (static provider-tool factory namespace). Settings: {baseURL?, apiKey?, headers?, fetch?, name?, generateId?} + Anthropic authToken (Bearer; mutually exclusive with apiKey -> InvalidArgumentError) + OpenAI organization/project/webSocket. baseURL: loadOptionalSetting(settingValue, envVar ANTHROPIC_BASE_URL/OPENAI_BASE_URL) then withoutTrailingSlash; Anthropic normalizes 'https://api.anthropic.com' -> '.../v1', default 'https://api.anthropic.com/v1'; OpenAI default 'https://api.openai.com/v1'. getHeaders is a closure re-evaluated per request: Anthropic {anthropic-version:'2023-06-01', x-api-key: loadApiKey(env ANTHROPIC_API_KEY) OR Authorization Bearer authToken, ...options.headers} wrapped by withUserAgentSuffix('ai-sdk/anthropic/VERSION'); OpenAI {Authorization: Bearer key(env OPENAI_API_KEY), OpenAI-Organization, OpenAI-Project}. options.name overrides provider id string ('anthropic.messages', '<name>.chat', '<name>.responses') and drives the providerOptions/providerMetadata namespace key (prefix before first '.').

### Model class + config shape
AnthropicLanguageModel(modelId, config) where config={provider:string, baseURL:string, headers?:Resolvable<Record<string,string|undefined>>, fetch?, buildRequestUrl?(baseURL,isStreaming), transformRequestBody?(args,betas), supportedUrls?():Record<string,RegExp[]>, generateId?, supportsNativeStructuredOutput?, supportsStrictTools?} — the two hooks exist so Bedrock/Vertex forks reuse the class (exported via internal/index.ts). OpenAI uses OpenAIConfig={provider, url:({modelId,path})=>string, headers?:()=>Record, fetch?, generateId?, fileIdPrefixes?}. Both classes expose: specificationVersion, modelId, provider getter, supportedUrls (OpenAI chat: {'image/*':[https regex]}; responses adds 'application/pdf'; Anthropic set from factory), doGenerate, doStream, plus WORKFLOW_SERIALIZE/DESERIALIZE statics for workflow persistence.

### getArgs pipeline (shared recipe)
Steps in both providers: 1) warnings:[SharedV4Warning] for unsupported params (Anthropic: frequencyPenalty/presencePenalty/seed unsupported, temperature clamped to [0,1]; OpenAI responses: topK/seed/presencePenalty/frequencyPenalty/stopSequences unsupported); 2) parseProviderOptions({provider:'anthropic'|'openai'|custom name, providerOptions, schema}) — Anthropic parses BOTH canonical 'anthropic' key and custom key then Object.assign merges (custom wins), and dual-publishes providerMetadata under both keys; OpenAI responses tries 'azure' key first if provider contains 'azure'; 3) getModelCapabilities(modelId) static substring-match table (Anthropic: maxOutputTokens 4096..128000, supportsStructuredOutput, supportsAdaptiveThinking, rejectsSamplingParameters, supportsXhighEffort, isKnownModel; OpenAI: isReasoningModel (o1/o3/o4-mini/gpt-5 except -chat), systemMessageMode 'developer' for reasoning models else 'system', supportsFlexProcessing, supportsPriorityProcessing, supportsNonReasoningParameters for gpt-5.1+); 4) top-level `reasoning` call option mapped to provider knobs (Anthropic thinking:{type:'adaptive'|'enabled',budgetTokens}|disabled + output_config.effort; OpenAI reasoning_effort / reasoning:{effort,summary,mode,context}); 5) prompt conversion; 6) tool preparation; 7) body assembly with conditional spreads. Anthropic getArgs also returns betas:Set<string> merged from prompt conversion, tool prep, user 'anthropic-beta' headers, and options.anthropicBeta; header 'anthropic-beta': Array.from(betas).join(',').

### Anthropic request body fields
POST {baseURL}/messages with {model, max_tokens (default from capabilities table; +thinkingBudget when thinking enabled; clamped to model max), temperature, top_k, top_p, stop_sequences, thinking?{type:'adaptive'|'enabled'|'disabled', budget_tokens?, display?}, output_config?{effort, task_budget, format:{type:'json_schema', schema:sanitizeJsonSchema(...)}}, speed?, inference_geo?, fallbacks?, cache_control?, metadata?{user_id}, mcp_servers?[], container? (string id or {id,skills[]}), system? (array of text blocks), messages[], context_management?{edits[]}, tools?, tool_choice?, stream:true|omitted}. When thinking enabled: temperature/top_k/top_p removed with warnings, default budget 1024 if missing. responseFormat json+schema: native output_config.format when structuredOutputMode 'outputFormat'/'auto'+supported, else synthesizes a `json` function tool with toolChoice required + disableParallelToolUse and converts the resulting tool_use into text content (isJsonResponseFromTool flips tool_use stop_reason back to 'stop').

### Anthropic prompt conversion
convertToAnthropicPrompt groups LanguageModelV4Prompt into blocks by role run: system|user(+tool)|assistant. First system block -> top-level `system` (array of {type:'text',text,cache_control}); later system blocks -> {role:'system'} message + beta 'mid-conversation-system-2026-04-07'. A user block merges consecutive user+tool messages into ONE user message whose content array holds text / image {source:{type:'base64'|'url'|'file'}} / document (pdf base64/url/file/plain text w/ title, context, citations{enabled}) / container_upload / tool_result {tool_use_id, content: string|array, is_error} parts. Assistant blocks merge into one assistant message: text (last block trailing-trimmed because API rejects trailing whitespace in prefill), compaction (text w/ providerOptions.anthropic.type==='compaction'), thinking {thinking, signature} / redacted_thinking {data} from reasoning parts' providerOptions (dropped w/ warning if sendReasoning=false or no signature), tool_use {id,name,input(object-wrapped via toAnthropicToolInput),caller?}, server_tool_use / mcp_tool_use / *_tool_result reconstruction for provider-executed tools (validated against per-tool output zod schemas). moveToolUseBlocksToEnd reorders tool_use after text within thinking-delimited segments. CacheControlValidator: reads providerOptions.anthropic.cacheControl|cache_control from part -> output -> message (last part only), max 4 breakpoints, warns and drops beyond; certain contexts canCache:false (thinking blocks). File parts with data.type 'reference' resolve via resolveProviderReference -> file_id + beta 'files-api-2025-04-14'; pdf beta 'pdfs-2024-09-25'.

### Anthropic tool preparation
prepareTools returns {tools?, toolChoice?, toolWarnings, betas}. Function tool -> {name, description, input_schema, cache_control?, eager_input_streaming?:true (default from stream && toolStreaming option, per-tool override providerOptions.anthropic.eagerInputStreaming), strict? (only if supportsStrictTools), defer_loading?, allowed_callers?, input_examples?}; strict tools add beta 'structured-outputs-2025-11-13', inputExamples/allowedCallers add 'advanced-tool-use-2025-11-20'. Provider tools switch on tool.id ('anthropic.computer_20250124' etc.) -> typed wire tool {type:'computer_20250124', name:'computer', display_width_px,...} each adding its beta (e.g. 'computer-use-2025-01-24', 'web-fetch-2025-09-10', 'code-execution-2025-08-25'); args validated via validateTypes against per-tool zod arg schemas; unknown id -> warning. toolChoice map: auto->{type:'auto'}, required->{type:'any'}, tool->{type:'tool',name}, none-> REMOVE tools entirely and send neither; all carry disable_parallel_tool_use. createToolNameMapping maps custom tool names <-> provider wire names (e.g. 'anthropic.web_search_20250305'->'web_search') and is used in both directions during response mapping.

### Anthropic SSE event -> stream part mapping
Chunk union (anthropicChunkSchema): message_start{message:{id,model,usage{input_tokens,cache_*},content? (pre-populated tool_use for programmatic calling), stop_reason?, container?}} -> 'response-metadata' + seed usage (+ synthesized tool-input-start/delta/end + tool-call for pre-populated tool_use); content_block_start{index, content_block: text|thinking|redacted_thinking|compaction|tool_use|server_tool_use|mcp_tool_use|mcp_tool_result|web_fetch_tool_result|web_search_tool_result|code_execution_tool_result|bash_code_execution_tool_result|text_editor_code_execution_tool_result|tool_search_tool_result|advisor_tool_result|fallback} -> text-start(id=String(index)) / reasoning-start / tool-input-start(id=tool_use.id) and registers contentBlocks[index] state {type, toolCallId, toolName, input:'', firstDelta, providerExecuted?, providerToolName?, providerToolInputType?, caller?}; *_tool_result blocks emit 'tool-result' (+ 'source' parts for web_search results) immediately at block start; 'fallback' dropped. content_block_delta{index, delta: text_delta{text}->text-delta, thinking_delta{thinking}->reasoning-delta, signature_delta{signature}->reasoning-delta(delta:'' + providerMetadata.anthropic.signature) only while blockType==='thinking', compaction_delta{content}->text-delta, input_json_delta{partial_json}->tool-input-delta (accumulated into contentBlock.input; firstDelta may inject '{"type": "<subtool>",' prefix for code-execution-20250825 subtools; empty deltas skipped), citations_delta{citation}->'source' part}. content_block_stop{index} -> text-end / reasoning-end / tool-input-end + final 'tool-call' with accumulated input ('' -> '{}'; code_execution inputs with {code} but no type get type:'programmatic-tool-call' injected). message_delta{delta:{stop_reason,stop_sequence,stop_details,container},usage{output_tokens,...iterations}} -> update finishReason/usage/metadata (no emit). message_stop -> 'finish' {finishReason, usage: convertAnthropicUsage, providerMetadata.anthropic {usage, stopSequence, stopDetails, iterations, container, contextManagement}}. ping -> ignored. error{error{type,message}} -> 'error' part. First-chunk protocol: transformedStream.tee(); read past stream-start (and optional raw); if first real chunk is 'error' throw APICallError(statusCode: type==='overloaded_error'?529:500, isRetryable: overloaded) because Anthropic returns HTTP 200 for overloads.

### Anthropic finish/usage mapping
mapAnthropicStopReason: pause_turn|end_turn|stop_sequence->'stop'; refusal->'content-filter'; tool_use->(isJsonResponseFromTool?'stop':'tool-calls'); max_tokens|model_context_window_exceeded->'length'; compaction|default->'other'. FinishReason is {unified, raw}. convertAnthropicUsage -> LanguageModelV4Usage {inputTokens:{total:input+cacheCreation+cacheRead, noCache:input, cacheRead, cacheWrite}, outputTokens:{total, text:undefined, reasoning:undefined}, raw}; when usage.iterations present (compaction/advisor) sums executor iterations (type 'compaction'|'message') unless a 'fallback_message' iteration exists (then top-level used as-is); advisor_message excluded from totals.

### OpenAI Chat API model
POST {baseURL}/chat/completions. Body: {model, logit_bias, logprobs/top_logprobs, user, parallel_tool_calls, max_tokens, temperature, top_p, frequency_penalty, presence_penalty, response_format ({type:'json_schema', json_schema:{schema,strict(default true),name(default 'response'),description}} | {type:'json_object'}), stop, seed, verbosity, max_completion_tokens, store, metadata, prediction, reasoning_effort, service_tier, prompt_cache_key/options/retention, safety_identifier, messages, tools, tool_choice}. Reasoning models: strip temperature/top_p/logprobs (unless effort==='none' && gpt-5.1+), frequency/presence penalty, logit_bias, top_logprobs; move max_tokens->max_completion_tokens. Message conversion: system -> role 'system'|'developer'|removed per systemMessageMode; user single text -> plain string content else parts [{type:'text'}|{type:'image_url',image_url:{url:dataURI|url,detail}}|{type:'input_audio',{data,format:'wav'|'mp3'}}|{type:'file',{file_id | filename+file_data dataURI}}]; assistant -> {content: string|null|textParts, tool_calls:[{id,type:'function',function:{name,arguments:JSON.stringify(input??{})}}]}; tool role -> one {role:'tool', tool_call_id, content:string} per result (json outputs stringified). promptCacheBreakpoint providerOption ({mode:'explicit'}) forces array-form content with prompt_cache_breakpoint on parts. prepareChatTools: function only -> {type:'function',function:{name,description,parameters,strict?}}; toolChoice auto|none|required pass through, tool->{type:'function',function:{name}}. doGenerate maps choices[0].message: content->text, tool_calls->tool-call (id ?? generateId()), annotations(url_citation)->source parts; providerMetadata.openai {acceptedPredictionTokens, rejectedPredictionTokens, logprobs}. doStream: body {stream:true, stream_options:{include_usage:true}}; chunk = {choices[0]:{delta:{content, tool_calls[{index,id,function:{name,arguments}}], annotations}, finish_reason, logprobs}, usage} | errorChunk; text uses single fixed id '0' with lazy text-start and text-end in flush; tool call deltas go through StreamingToolCallTracker (provider-utils) which emits tool-input-start/delta/end + tool-call keyed by index; finish emitted in flush. throwIfOpenAIStreamErrorBeforeOutput tees the stream and throws APICallError if an error frame arrives before any output chunk (status inferred from code/type keywords: rate_limit->429, authentication->401 etc.). mapOpenAIFinishReason: stop->stop, length->length, content_filter->content-filter, function_call|tool_calls->tool-calls, default->other.

### OpenAI Responses API model (default languageModel)
POST {baseURL}/responses. Body: {model, input: OpenAIResponsesInput (typed item list, not messages), temperature, top_p, max_output_tokens, text:{format:{type:'json_schema',strict,name,description,schema}|{type:'json_object'}, verbosity}, conversation, max_tool_calls, metadata, parallel_tool_calls, previous_response_id, store (API default true), user, instructions, service_tier, include:[...], prompt_cache_key/options/retention, safety_identifier, top_logprobs, truncation, context_management:[{type,compact_threshold}], reasoning:{effort,summary,mode,context}, tools, tool_choice, stream?}. Auto-include logic: logprobs->'message.output_text.logprobs'; web_search tool->'web_search_call.action.sources'; code_interpreter->'code_interpreter_call.outputs'; store===false && reasoning model->'reasoning.encrypted_content'. Input conversion: system/developer messages; user content -> input_text | input_image {image_url dataURI/url | file_id, detail} | input_file {file_id | file_url | filename+file_data}; assistant text -> {role:'assistant',content:[{type:'output_text',text}],id,phase?} OR {type:'item_reference',id} when store&&itemId (itemId from providerOptions.openai.itemId round-trip); reasoning parts -> {type:'reasoning', id?, encrypted_content?, summary:[{type:'summary_text',text}]} merged by itemId, or item_reference when store; client function tool-call -> {type:'function_call', call_id, name, arguments} (NEVER item_reference — call_id/fc_ id mismatch breaks pairing); provider tools -> typed items (local_shell_call, shell_call, apply_patch_call, custom_tool_call, tool_search_call) which MAY be item_reference; tool role -> function_call_output {call_id, output: string | array of input_text/input_image/input_file} / custom_tool_call_output / shell_call_output / local_shell_call_output / apply_patch_call_output / tool_search_output; tool-approval-response -> item_reference + {type:'mcp_approval_response', approval_request_id, approve}; compaction custom part -> {type:'compaction', id, encrypted_content}. ### Warnings and metadata channels
SharedV4Warning union used everywhere: {type:'unsupported',feature,details?} | {type:'compatibility',feature,details} | {type:'deprecated',setting,message} | {type:'other',message}; collected during getArgs+conversion+tool prep, emitted as first stream part {type:'stream-start',warnings} and in doGenerate result. providerOptions (input) and providerMetadata (output) are Record<providerName, Record<string,JSONValue>>; roundtripping is load-bearing: Anthropic reasoning signature/redactedData, cacheControl; OpenAI itemId/reasoningEncryptedContent/phase/annotations/namespace/approvalRequestId must survive a round trip through the app layer to reconstruct wire history. doGenerate result shape: {content:[LanguageModelV4Content], finishReason:{unified,raw}, usage, request:{body}, response:{id?,timestamp?,modelId?,headers,body}, warnings, providerMetadata}; doStream result: {stream, request:{body}, response:{headers}}.

### Response schema philosophy
All wire schemas are deliberately partial ('limited version of the schema, focused on what is needed') with .nullish() on response fields and looseObject/passthrough for usage — unknown fields must be tolerated, and the Responses chunk schema catches unrecognized event types as {type:'unknown_chunk'} rather than failing. Provider options schemas are strict/optional by contrast. Error responses: Anthropic {type:'error',error:{type,message}}; OpenAI {error:{message,type?,param?,code?}}. CLAUDE.md mandates never raw JSON.parse (secureJsonParse/safeParseJSON to avoid proto pollution).

## Port implications
A Zig provider is precisely: (1) a Config struct {provider_name:[]const u8, base_url, headers: fn or resolvable map, http_client/fetch injection, optional hooks buildRequestUrl/transformRequestBody}; (2) a factory fn taking Settings {base_url?, api_key?, headers?, name?} with env-var fallback (std.process.getEnvVarOwned) and trailing-slash normalization; (3) a LanguageModel vtable impl (comptime duck-typing or explicit vtable) with doGenerate/doStream. Natural mappings: every SSE chunk union and stream-part union becomes a Zig tagged union — the code is already written as exhaustive discriminated-union switches (TS `never` checks), so it ports almost mechanically. The doStream TransformStream is a state machine with captured mutable state (finishReason, usage accumulator, per-index/per-output_index hashmaps of in-flight blocks, reasoning summary lifecycle enum {active,can_conclude,concluded}) — in Zig this is a struct with a `transform(chunk) -> []StreamPart` method fed by an SSE line parser over std.Io reader; no TransformStream machinery needed, but the stream tee()/first-chunk-error-peek pattern (Anthropic 200-overload, OpenAI early-error) must become a buffered peek: read+map events into a small queue before handing the stream to the consumer, replaying queued parts. Hard parts: (a) zod schema replacement — need a tolerant JSON decoder that ignores unknown fields, treats null==missing (nullish), and coerces unknown Responses event types to an `unknown` variant instead of erroring; std.json with ignore_unknown_fields covers most, but unions-by-'type'-field need a custom dispatch; (b) providerOptions/providerMetadata are open string-keyed JSON maps namespaced by provider name — model as std.json.Value maps plus typed parse helpers per provider (parseProviderOptions equivalent), with round-trip preservation (itemId, encrypted reasoning, signatures) as an invariant; (c) dynamic body assembly with conditional-field spreads maps to a json.WriteStream builder or struct-with-optionals + emit_null_optional_fields=false; (d) manual JSON-string tool-input synthesis (code_interpreter/apply_patch prefix deltas, escapeJSONDelta = JSON-escape without quotes) must be replicated byte-exactly; (e) betas Set -> StringHashMap accumulated across prompt/tool/option passes, joined into one header; (f) memory: per-request arena is ideal since request body, converted prompt, and accumulated tool-input strings share a lifetime, but stream state (accumulating tool input strings across events) needs an allocator owned by the stream object freed on finish/cancel. FFI layer should expose the stream as a pull-based next_part() over the tagged-union StreamPart serialized to a stable C ABI struct or JSON.
