# sdk-inventory

## Summary
The monorepo at inspiration/packages/ has 70 packages (pnpm workspace: packages/*, tools/*, examples/*). The dependency spine is explicitly documented in AGENTS.md: ai -> @ai-sdk/provider-utils -> @ai-sdk/provider, with every provider also -> provider-utils -> provider. Crucially, `ai` also has a hard runtime dependency on @ai-sdk/gateway (string model ids like 'anthropic/claude-opus-4.6' resolve through the Vercel AI Gateway via ai/src/model/resolve-model.ts and global.ts), so the minimal CORE port order is: provider (225 files/9,549 LOC, pure interface specs — LanguageModelV4 current plus legacy v2/v3, embedding/image/speech/transcription/video/realtime/reranking models, middleware variants, JSON value types, AISDKError marker-pattern errors) → provider-utils (142/7,941: fetch abstraction, post/get-to-api, response handlers, parse-json-event-stream SSE parser, secure JSON parse, schema layer with vendored zod3-to-json-schema, retry-with-exponential-backoff, websocket, uint8 utils, id generation) → gateway (47/4,905: one gateway model per modality + auth + metadata fetch) → ai (321/36,475: generateText/streamText/generateObject, ToolLoopAgent, embed/rerank/image/speech/video/transcribe/realtime, prompt conversion, registry, middleware, telemetry hooks, ui-message-stream protocol, framework-agnostic Chat in src/ui). openai-compatible (25/2,875) is the base class for ~12 thin providers. 40 packages are concrete providers; 5 are UI-framework bindings (react/vue/svelte/angular/rsc); harness/workflow/sandbox/policy-opa form an experimental agent-runtime layer; codemod/devtools/test-server/otel/tui/langchain/llamaindex/valibot are tooling/adapters. Full per-package table in key_findings.

## Key findings
### CORE dependency order (port order)
1) @ai-sdk/provider (no deps) 2) @ai-sdk/provider-utils (-> provider) 3) @ai-sdk/gateway (-> provider, provider-utils; HARD dependency of ai, not optional — ai/src/index.ts, ai/src/model/resolve-model.ts, ai/src/types/language-model.ts, ai/src/prompt/wrap-gateway-error.ts import @ai-sdk/gateway) 4) ai (-> gateway, provider, provider-utils). Parallel after step 2: @ai-sdk/openai-compatible (-> provider, provider-utils) which is the parent of alibaba, baseten, cerebras, deepinfra, fireworks, huggingface, moonshotai, togetherai, vercel, xai, google-vertex(partial); @ai-sdk/mcp (-> provider, provider-utils); all native providers (-> provider, provider-utils only). Wrapper providers: azure -> openai+deepseek; anthropic-aws -> anthropic; amazon-bedrock -> anthropic+openai; google-vertex -> google+anthropic+openai-compatible. test-server has zero workspace deps.

### CORE: provider
@ai-sdk/provider — pure specification package, zero deps. 225 src files, 9,549 LOC. Dirs: language-model (v2/, v3/, v4/ — current spec is LanguageModelV4 with LanguageModelV4CallOptions, -Prompt, -ToolChoice etc.), embedding-model, image-model, speech-model, transcription-model, video-model, realtime-model, reranking-model, *-middleware dirs, provider (ProviderV~ registry iface), errors (AISDKError base + ~17 subclasses using Symbol.for('vercel.ai.error.AI_<Name>') marker pattern for isInstance checks), json-value, files, shared, skills. This is the Zig-port foundation: mostly type shapes, no runtime logic.

### CORE: provider-utils
@ai-sdk/provider-utils — 142 src files, 7,941 LOC. Runtime plumbing shared by core and all providers: fetch-function.ts (injectable fetch), post-to-api.ts/get-from-api.ts, response-handler.ts (JSON + streaming SSE response handlers), parse-json-event-stream.ts (SSE parsing), extract-lines, secure-json-parse (mandatory instead of JSON.parse per repo rules — parseJSON/safeParseJSON), schema.ts (Schema abstraction bridging zod3/zod4/standard-schema), to-json-schema/ (vendored zod3-to-json-schema), validate-types, retry-with-exponential-backoff, delay/delayed-promise, generate-id, uint8-utils, websocket.ts, load-api-key/load-setting, combine/normalize headers, detect-media-type, download-blob, streaming-tool-call-tracker, convert-async-iterator-to-readable-stream. This is where the Zig std.Io/HTTP/streaming heart lives.

### CORE: ai
Package `ai` (npm name ai) — 321 src files, 36,475 LOC. src/ dirs: generate-text (incl. streamText loop, tool calling, step control), generate-object, agent (ToolLoopAgent), embed, rerank, generate-image, generate-speech, generate-video, transcribe, realtime, prompt (message conversion, standardization), model (resolve-model: string id -> gateway model), registry (provider registry), middleware, telemetry (hooks; actual otel exporter lives in @ai-sdk/otel), text-stream, ui-message-stream (the SSE wire protocol between server and UI clients), ui (framework-agnostic Chat class, ChatTransport interface, default-chat-transport/http-chat-transport/direct-chat-transport/text-stream-chat-transport, process-ui-message-stream, convert-to-model-messages, use-completion core), upload-file, upload-skill, logger, error, types, util, test. The ui/ subdir is framework-agnostic state (react/vue/svelte/angular are thin bindings over it) so it IS portable core.

### CORE: gateway
@ai-sdk/gateway — 47 src files, 4,905 LOC. Vercel AI Gateway provider: gateway-language-model, -embedding-model, -image-model, -speech-model, -transcription-model, -video-model, -realtime-model (incl. gateway-realtime-auth), -reranking-model, gateway-fetch-metadata (available-models discovery), gateway-config/headers, errors/. Must be ported (or at least stubbed to interface level) before `ai` because ai imports it directly for default string-model resolution.

### TRANSPORT/PROTOCOL packages
mcp (@ai-sdk/mcp, 20 files/5,286 LOC): MCP client (tool/mcp-client.ts), json-rpc-message.ts, transports: mcp-http-transport.ts, mcp-sse-transport.ts, mcp-stdio/ (child-process spawn), oauth.ts + oauth-util (full OAuth flow), mcp-apps.ts; depended on by react. open-responses (9/2,218): provider for openresponses.org-compatible APIs (OpenAI Responses protocol clone) — PROVIDER-shaped but protocol-generic. openai-compatible (25/2,875): OpenAI Chat Completions/embedding/image protocol base reused by ~12 provider packages — port early, right after provider-utils. gateway also fits here but is classified CORE due to the hard ai dependency.

### UI-FRAMEWORK packages

All are thin reactive-state bindings over the framework-agnostic Chat/transport machinery that lives in ai/src/ui — the logic worth porting is already counted in CORE `ai`; these packages themselves are out of scope for Zig (a C-ABI consumer could rebuild equivalents).

### Agent-runtime layer (experimental)
harness (58f/9,612: HarnessV1 spec + HarnessAgent + sandbox interface + bridge; deps provider, provider-utils, ai), adapter packages harness-claude-code (9/2,724), harness-codex (11/2,727), harness-deepagents (10/1,659), harness-opencode (13/3,879), harness-pi (16/3,596) — each adapts an external coding-agent runtime to HarnessV1. sandbox-just-bash (4/455) and sandbox-vercel (4/754) implement HarnessV1SandboxProvider. workflow (15/5,890: durable WorkflowAgent over Vercel Workflow — do-stream-step, workflow-chat-transport, serializable-schema), workflow-harness (3/762). policy-opa (12/874: OPA/.rego policy-as-code toolApproval hook for tool calls). All marked experimental upstream; classify TOOLING/INFRA / deferred — depends on CORE ai, not depended on by it.

### TOOLING/INFRA and OUT-OF-SCOPE
codemod (323f/13,828: jscodeshift TS migration scripts — OUT-OF-SCOPE). devtools (8/2,420: local web UI inspecting SDK requests — OUT-OF-SCOPE). test-server (4/354: create-test-server.ts vitest-integrated HTTP mock server + convert-array-to-readable-stream; used as devDependency by nearly every package's tests — the CONCEPT must be recreated in Zig to port the test suites). otel (15/3,975: OpenTelemetry span recording for ai — depends on ai + @opentelemetry/api; port later, optional). tui (11/3,192: terminal agent UI over ai — interesting future Zig target but not core). langchain (adapter ai<->LangChain streams; real code ~2.9k LOC, plus 16.5k-LOC test fixture — OUT-OF-SCOPE). llamaindex (3/117 adapter — OUT-OF-SCOPE). valibot (2/17: valibotSchema wrapper — OUT-OF-SCOPE, JS-schema-ecosystem specific).

### Workspace/build facts
pnpm-workspace.yaml: packages/*, tools/*, examples/*, packages/rsc/tests/e2e/next-server. Root README.md is a symlink to packages/ai/README.md. Build: turborepo; tests: vitest with node+edge runs; lint oxlint/oxfmt. Repo rule: never JSON.parse directly — use parseJSON/safeParseJSON from provider-utils (security). Error convention: AISDKError subclasses with Symbol.for('vercel.ai.error.AI_<Name>') markers. Provider schema conventions: user-facing options use .optional(), API response schemas use .nullish() and stay minimal. ADRs in contributing/decisions/, architecture walkthrough in architecture/provider-abstraction.md.

## Port implications
Port order is forced by the dependency spine: (1) provider — almost pure type declarations; maps naturally to Zig tagged unions/structs per LanguageModelV4 content parts, an error set + error-payload struct hierarchy replacing the AISDKError marker pattern, and vtable-style interfaces (fn tables or anytype/comptime duck typing) for the 8 model modalities plus middleware wrappers. Port only V4, skip legacy v2/v3 dirs. (2) provider-utils — the real concurrency core: injectable fetch (std.http.Client behind a function-pointer table for FFI overrides), SSE line/event parsing as std.Io.Reader streaming, retry/backoff, secure JSON parsing (std.json with depth/size limits), and a Schema abstraction where zod must be replaced by comptime-generated JSON Schema from Zig types plus a dynamic JSON-Schema value for the C ABI. (3) gateway before ai — ai hard-imports it for string-model resolution; either port it fully or make resolve-model pluggable. (4) ai — largest surface (36.5k LOC); streamText's multi-step tool loop, ui-message-stream SSE protocol, and the framework-agnostic Chat/ChatTransport in ai/src/ui are all portable and should be ported (they are what Python/Rust wrappers will bind). ReadableStream/AsyncIterator semantics must map to std.Io async + reader/writer pipes; AbortSignal maps to cancellation tokens. Providers: port openai-compatible early to unlock 12 thin providers nearly for free; big native providers (openai, anthropic, google) are mostly zod-schema translation + stream-part mapping. Recreate test-server as a Zig in-process HTTP mock to port the vitest suites. Defer/skip: react/vue/svelte/angular/rsc, codemod, devtools, langchain/llamaindex, valibot, harness/workflow family, otel (later, optional), tui (possible future showcase). MCP is in scope (stdio child-process, SSE, HTTP transports map well to std.process/std.Io).
