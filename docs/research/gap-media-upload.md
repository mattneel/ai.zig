# gap:Media/file orchestration: generateImage, generateSpeech, transcribe, generateVideo, uploadFile/uploadSkill + one concrete media provider

## Summary
Surveyed all six media-generation entry points in packages/ai/src (generate-image, generate-speech, transcribe + stream-transcribe, generate-video, upload-file, upload-skill), their result types and errors, plus the OpenAI image/speech/transcription provider models end-to-end and the provider-utils HTTP layer (postToApi/postFormDataToApi/convertToFormData/response handlers/retry/detectMediaType/download). Pattern: every generate* fn resolves the model, suffixes User-Agent (`ai/${VERSION}`), builds a retry closure via prepareRetries (default maxRetries=2, exponential backoff 2000ms x2, honoring retry-after-ms/retry-after headers, abort-aware), normalizes DataContent (string base64 / data-URL / http URL / Uint8Array / ArrayBuffer) into tagged provider file unions, fans out parallel doGenerate calls when n > maxImagesPerCall/maxVideosPerCall, merges warnings/usage/providerMetadata/responses, logs warnings once, throws AI_No{Image,Speech,Transcript,Video}GeneratedError({responses}) on empty output, and returns a result class with lazy base64/Uint8Array conversion (DefaultGeneratedFile). Video polling is NOT in core — it lives inside provider doGenerate (xAI: create job POST → while(true){delay(pollIntervalMs=5000); timeout 600000ms; GET status; done/expired/failed/pending}). OpenAI transcription uses multipart/form-data via native FormData handed directly to fetch (boundary auto-generated by runtime); image edits use convertToFormData with Blob parts; speech returns raw binary bytes via createBinaryResponseHandler. Zig 0.16 std.http has NO multipart encoder but its BodyWriter (content_length | chunked states over *std.Io.Writer with drain/sendFile vtables) fully supports streaming multipart bodies; the port needs a small multipart encoder that writes boundary-delimited parts to BodyWriter.writer.

### Image input normalization (toImageModelV4File)

### Retry machinery
prepareRetries (ai/src/util/prepare-retries.ts): validates integer >= 0 else InvalidArgumentError, default 2. retryWithExponentialBackoffRespectingRetryHeaders: initialDelayInMs=2000, backoffFactor=2; shouldRetry = APICallError.isInstance(e) && e.isRetryable===true (or GatewayError); getDelayInMs prefers 'retry-after-ms' header (float ms) then 'retry-after' (seconds or HTTP date), accepted only if 0<=ms && (ms<60000 || ms<exponentialBackoffDelay), else exponential delay. Core retryWithExponentialBackoff (provider-utils): abort errors rethrown immediately; maxRetries===0 -> raw error; recursive with delayInMs*=backoffFactor; tryNumber>maxRetries -> RetryError maxRetriesExceeded; non-retryable on first try -> raw error, later tries -> RetryError errorNotRetryable. ### Zig std.http BodyWriter (0.16) — streaming request bodies
lib/std/http.zig:741 pub const BodyWriter = struct { http_protocol_output: *Writer, state: State, writer: Writer }; State = union(enum){ none, content_length: u64, chunk_len: usize, end }. writer is a std.Io.Writer whose vtable is chosen by Client.Request.sendBodyUnflushed(r, buffer) per r.transfer_encoding (union {content_length: u64, chunked, none}): contentLengthDrain/contentLengthSendFile decrement state.content_length (end() asserts it reaches 0); chunkedDrain writes 8-hex-digit chunk headers ('00000000\r\n' template patched via writeHex) sized to the full buffered+splat data, then payload, '\r\n', with endChunked/endChunkedUnflushed emitting '0\r\n' + optional trailers + '\r\n'; noneDrain writes raw. chunkedSendFile requires a computable file size (Writer.countSendFileLowerBound) else error.Unimplemented — chunk header must precede data. Request.sendBody(buffer) flushes head then returns BodyWriter; sendBodyComplete(body: []u8) is the one-shot content-length path. Client.fetch only takes payload: ?[]const u8 (in-memory), so streaming multipart must use client.request() + sendBody. Headers: RequestOptions{headers: Request.Headers{content_type: .{.override = "..."}}, extra_headers: []const http.Header}. There is NO multipart encoder anywhere in std (grep 'multipart|boundary' over lib/std: zero hits outside HeaderIterator comment); ziex only has a server-side multipart parser (src/runtime/server/Action.zig checks content-type contains 'multipart/form-data'), no encoder.

### Multipart encoder requirements for Zig port
The port must implement a FormData-equivalent encoder writing to BodyWriter.writer (*std.Io.Writer). Wire format needed by OpenAI transcription/image-edit: for each text field `--{boundary}\r\nContent-Disposition: form-data; name="{name}"\r\n\r\n{value}\r\n`; for file parts `--{boundary}\r\nContent-Disposition: form-data; name="file"; filename="audio.{ext}"\r\nContent-Type: {mediaType}\r\n\r\n{bytes}\r\n`; terminator `--{boundary}--\r\n`. Boundary: random token (browsers use '----WebKitFormBoundary' + 16 alnum chars); Content-Type request header must be `multipart/form-data; boundary={boundary}` set via Request.Headers.content_type override. Two transfer strategies both supported by std: (1) exact Content-Length precomputed — always possible in this SDK because all part payloads are in-memory []u8 (audio Uint8Array, decoded base64 images); length = sum over parts of (2+boundary.len+2 + disposition/type header bytes + 2 + data.len + 2) + boundary.len+6; (2) transfer_encoding=.chunked for genuinely streaming sources (BodyWriter handles chunk framing transparently). Array fields must support repeated appends under `key[]` names (timestamp_granularities[]) matching convertToFormData semantics (single-element arrays use the bare key).

## Port implications
Natural Zig mappings: DataContent -> tagged union { base64: []const u8, bytes: []const u8, url: []const u8, arraybuffer collapses into bytes }; normalization functions become pure fns returning a File union { url: []const u8, file: struct { media_type, data } } mirroring ImageModelV4File/VideoModelV4File; SharedV4FileData is a clean 4-variant tagged union. The GeneratedFile lazy base64<->bytes caching maps to a struct with optional fields plus getter methods taking an allocator (decide ownership: recommend eager bytes + on-demand base64 into caller-provided allocator). Batching (n > maxImagesPerCall) uses Promise.all — in Zig std.Io this is the concurrency heart: spawn one task per call (Io.async/concurrent), each running the retry wrapper (delay must honor cancellation like delay(ms, {abortSignal}); Zig side: Io timeout/sleep + cancel). Retry policy is a small state machine: maxRetries default 2, initialDelay 2000ms, factor 2, retry only when APICallError.isRetryable, header-derived delay from retry-after-ms/retry-after capped at 60s. Video generation's deferred-job polling lives in provider code (create -> sleep(pollInterval) -> GET status loop with wall-clock timeout), so providers need cancellable sleep + repeated GETs on one client. Error hierarchy: AISDKError marker pattern collapses to Zig error sets + a rich diagnostic struct (Zig errors carry no payload; return a Result-style union or out-param diagnostics with responses metadata, as NoImageGeneratedError carries responses[]). Hard parts: (1) multipart — std.http has no encoder; write one over BodyWriter.writer emitting boundary-delimited parts; since all payloads are in-memory, precompute exact Content-Length and use transfer_encoding=.content_length (avoids chunked pitfalls; chunkedSendFile fails on unknown-size files); must support repeated `key[]` fields and filename/Content-Type per part; set Content-Type via Request.Headers.content_type = .{ .override = "multipart/form-data; boundary=..." }. (2) Binary responses (speech): read body reader to ArrayList — trivial. (3) streamTranscribe requires WebSockets (not in std) and a TransformStream/DelayedPromise equivalent (futures + bounded queue over std.Io) — schedule it after the HTTP-only functions. (4) detectMediaType is a pure magic-bytes table — direct port, including base64-prefix decode of first 24 chars and ID3 skip. C FFI layer: result objects are plain data (bytes + strings + arrays of warnings/metadata) so they serialize cleanly; expose provider metadata as JSON strings to avoid modeling arbitrary JSONObject in the ABI.
