# sdk-streams-ui-mcp

## Summary
Read all of packages/ai/src/ui-message-stream, text-stream, ui, packages/mcp/src, and the tool machinery (which lives in @ai-sdk/provider-utils, not packages/ai). Findings: (1) The UI message wire protocol is a ~27-variant discriminated union (UIMessageChunk) serialized as SSE lines `data: {json}\n\n` terminated by `data: [DONE]\n\n`, with fixed response headers including `x-vercel-ai-ui-message-stream: v1`. (2) Server-side stream construction (createUIMessageStream with a writer supporting write/merge/onError, handleUIMessageStreamFinish for message-id injection and onStepEnd/onEnd callbacks, toUIMessageChunk mapping streamText's TextStreamPart to chunks) is pure logic plus WHATWG streams — fully portable. (3) The client-side reducer processUIMessageStream folds chunks into a UIMessage (parts array with text/reasoning/tool/data/source/file parts and 7-state tool invocation lifecycle incl. approvals), using incremental partial-JSON parsing for tool-input deltas; AbstractChat is a framework-agnostic chat state machine (status submitted/streaming/ready/error, SerialJobExecutor, AbortController, transports); React hooks live only in packages/react and wrap AbstractChat via useSyncExternalStore — not portable. (4) convertToModelMessages and validateUIMessages are pure conversions/validations — portable. (5) The MCP package is a full JSON-RPC 2.0 client with id-correlated request/response over three transports (stdio child process with newline-delimited JSON; HTTP+SSE legacy; Streamable HTTP with session ids, inbound resumable SSE, reconnection backoff) plus OAuth 2.1 discovery/exchange, and converts MCP tools to SDK dynamicTool/tool values whose execute calls tools/call. (6) tool()/dynamicTool() are identity/tagging functions over a Tool data record (inputSchema, execute, onInputStart/onInputDelta/onInputAvailable, toModelOutput, needsApproval) — all runtime behavior lives in the generate-text loop.

## Key findings
### UIMessageChunk wire protocol (ui-message-chunks.ts)
Discriminated union on `type` with variants: text-start/text-delta/text-end (id, delta, providerMetadata?), reasoning-start/-delta/-end, error{errorText}, tool-input-start{toolCallId,toolName,providerExecuted?,toolMetadata?,dynamic?,title?}, tool-input-delta{toolCallId,inputTextDelta}, tool-input-available{...,input}, tool-input-error{...,errorText}, tool-approval-request{approvalId,toolCallId,isAutomatic?,signature?}, tool-approval-response{approvalId,approved,reason?}, tool-output-available{toolCallId,output,preliminary?}, tool-output-error{toolCallId,errorText}, tool-output-denied{toolCallId}, custom{kind:'{provider}.{type}'}, source-url{sourceId,url,title?}, source-document{sourceId,mediaType,title,filename?}, file/reasoning-file{url,mediaType}, `data-{NAME}`{id?,data,transient?} (dynamic type key prefix), start-step, finish-step, start{messageId?,messageMetadata?}, finish{finishReason?,messageMetadata?}, abort{reason?}, message-metadata{messageMetadata}. A zod schema (uiMessageChunkSchema) validates inbound chunks on the client. finishReason enum: stop|length|content-filter|tool-calls|error|other.

### SSE encoding + response headers
JsonToSseTransformStream: each chunk -> `data: ${JSON.stringify(part)}\n\n`; on flush emits `data: [DONE]\n\n`. UI_MESSAGE_STREAM_HEADERS = {content-type: text/event-stream, cache-control: no-cache, connection: keep-alive, x-vercel-ai-ui-message-stream: v1, x-accel-buffering: no}. createUIMessageStreamResponse pipes chunk stream through SSE transform + TextEncoderStream into a Response; optional consumeSseStream callback receives a tee()'d copy (fire-and-forget, e.g. for redis persistence of resumable streams). pipeUIMessageStreamToResponse does the same onto node:http ServerResponse via writeToServerResponse (writeHead, backpressure via 'drain' event, optional response.flush(), response.end() in finally). Client side parses via parseJsonEventStream: bytes -> TextDecoderStream -> EventSourceParserStream (eventsource-parser dep) -> safeParseJSON per event, skipping literal '[DONE]'.

### createUIMessageStream (server-side composition)
Signature: ({execute({writer}), onError=()=>'An error occurred.', originalMessages?, onStepEnd?/onStepFinish?(deprecated), onEnd?/onFinish?(deprecated), generateId=generateIdFunc}) -> ReadableStream<UIMessageChunk>. Writer interface: write(part), merge(otherStream) (spawns an async reader loop pushing into shared controller; errors become error chunks via onError), onError. Concurrency: ongoingStreamPromises array; a loop awaits/shifts until empty (allows merges initiated after execute returned), then closes controller. safeEnqueue swallows enqueue-after-close. Default onError masks server errors from clients.

### handleUIMessageStreamFinish (id injection + persistence callbacks)
If last originalMessage is assistant, reuse its id (continuation); a TransformStream injects messageId into 'start' chunks lacking one and sets isAborted on 'abort'. If no onEnd/onStepEnd, returns id-injected stream unchanged (zero-cost). Otherwise runs processUIMessageStream with a no-op write to accumulate StreamingUIMessageState, calls onStepEnd (with structuredClone'd responseMessage + messages) on each 'finish-step' chunk, and onEnd({isAborted,isContinuation,responseMessage,messages,finishReason}) on flush AND cancel (guarded by finishCalled flag). getResponseUIMessageId: undefined when originalMessages==null (client generates ids); else lastMessage.id if assistant, else generateMessageId().

### toUIMessageChunk / toUIMessageStream (TextStreamPart -> UIMessageChunk)
Options {tools?, sendReasoning=true, sendSources=false, sendStart=true, sendFinish=true, onError, messageMetadata?, responseMessageId?}. Mapping: text-delta part.text->delta; file parts to data: URLs (`data:{mediaType};base64,{base64}`); source->source-url/source-document; tool-call -> tool-input-available (or tool-input-error when part.invalid, errorText=onError(part.error)); tool-result -> tool-output-available with output===undefined coerced to null (JSON drops undefined); tool-error -> tool-output-error (providerExecuted errors passed raw, else masked by onError); tool-input-end and raw parts dropped; isDynamic resolved from tools[toolName].type==='dynamic' or part.dynamic for unknown tools. toUIMessageStream additionally emits standalone message-metadata chunks when messageMetadata({part}) returns non-null for non start/finish parts, then wraps with handleUIMessageStreamFinish.

### text-stream/ (trivial, portable)
toTextStream: filter TextStreamPart for type==='text-delta', emit part.text strings. createTextStreamResponse: Response with content-type text/plain; charset=utf-8, UTF-8 encoded chunks. pipeTextStreamToResponse: same onto node ServerResponse. prepareHeaders merges user headers with defaults (user wins, case-insensitive Headers).

### UIMessage model (ui-messages.ts)
UIMessage{id, role:'system'|'user'|'assistant', metadata?, parts: UIMessagePart[]}. Parts union: TextUIPart{type:'text',text,state?:'streaming'|'done'}, ReasoningUIPart (same shape), CustomContentUIPart{kind}, ToolUIPart (type=`tool-{NAME}`) and DynamicToolUIPart (type='dynamic-tool', +toolName) — both carry toolCallId, title?, toolMetadata?, providerExecuted? plus a 7-state discriminated lifecycle: input-streaming(partial input) -> input-available -> approval-requested{approval:{id,isAutomatic?,signature?}} -> approval-responded{approval.approved:boolean} -> output-available{output,preliminary?} | output-error{errorText,rawInput?} | output-denied; SourceUrlUIPart, SourceDocumentUIPart, FileUIPart{mediaType,filename?,url,providerReference?}, ReasoningFileUIPart, DataUIPart (type=`data-{NAME}`,{id?,data}), StepStartUIPart{type:'step-start'}. Type guards: isToolUIPart = type.startsWith('tool-') || type==='dynamic-tool'; getStaticToolName = type.split('-').slice(1).join('-').

### processUIMessageStream (client reducer, the protocol's core state machine)
StreamingUIMessageState{message, activeTextParts: Record<id,TextUIPart>, activeReasoningParts, partialToolCalls: Record<toolCallId,{text,index,toolName,dynamic?,title?,toolMetadata?}>, finishReason?}. Runs each chunk through runUpdateMessageJob (serialized via SerialJobExecutor in Chat) mutating state.message.parts in place and calling write() to publish snapshots. tool-input-delta accumulates raw text and re-parses with parsePartialJson on every delta to produce partial input objects. Deltas without a prior start chunk throw UIMessageStreamError (chunkType, chunkId). start chunk may override message.id; message-metadata merged via mergeObjects + optional schema validation (validateTypes). data-* chunks: optional per-type schema validation; transient ones only invoke onData and are not stored; non-transient reconcile by (type,id) replacing .data in place. finish-step clears active text/reasoning maps; error chunk invokes onError only. onToolCall invoked (blocking) for tool-input-available of non-provider-executed tools. Maps use Object.create(null) (createIdMap) to avoid prototype pollution from untrusted ids.

### convertToModelMessages (UIMessage[] -> ModelMessage[])
async fn(messages, {tools?, ignoreIncompleteToolCalls?, convertDataPart?}). system: concat text parts. user: text/file parts (file data as {type:'url'} or {type:'reference'} when providerReference present); data parts via convertDataPart callback. assistant: parts accumulated into blocks split at 'step-start'; each block emits an assistant message (text, custom, file, reasoning, reasoning-file, tool-call entries with input; tool-approval-request entries; inline tool-result for providerExecuted output states via createToolModelOutput) followed by a 'tool' role message containing ToolResultPart/ToolApprovalResponse entries for non-provider-executed tools with outputs, synthetic {type:'execution-denied'} results for denied approvals, and {type:'error-text'} for output-denied. Uses tool.toModelOutput via createToolModelOutput(getOwn(tools,toolName)). Throws MessageConversionError on unknown role.

### AbstractChat + ChatTransport (framework-agnostic client)
AbstractChat<UI_MESSAGE> holds ChatState{status:'submitted'|'streaming'|'ready'|'error', error, messages, pushMessage, popMessage, replaceMessage, snapshot} (state impl provided by framework wrapper; React's Chat in packages/react uses useSyncExternalStore + structuredClone snapshot). Methods: sendMessage (text/files or parts; optional messageId replaces a user message and truncates after), regenerate, resumeStream, stop (aborts AbortController), clearError, addToolOutput({tool,toolCallId,output|errorText}), addToolApprovalResponse({id,approved,reason?}) — the latter two run on SerialJobExecutor and may auto-resubmit via sendAutomaticallyWhen (helpers: lastAssistantMessageIsCompleteWithToolCalls / ...WithApprovalResponses). makeRequest: transport.sendMessages({chatId,messages,abortSignal,trigger:'submit-message'|'regenerate-message'|'resume-stream',messageId,headers,body,metadata}) -> ReadableStream<UIMessageChunk> consumed through processUIMessageStream; write() sets status streaming and push/replace last message; disconnect detection via TypeError message containing 'fetch'/'network'. ChatTransport iface = {sendMessages, reconnectToStream(chatId)->stream|null}. HttpChatTransport: POST JSON {id, messages, trigger, messageId, ...body} to api ('/api/chat' default) with prepareSendMessagesRequest hook; reconnect: GET `{api}/{chatId}/stream`, 204 => null. DefaultChatTransport parses SSE + validates with uiMessageChunkSchema; TextStreamChatTransport wraps plain text via transformTextToUiMessageStream (synthesizes start/start-step/text-start id 'text-1'/deltas/text-end/finish-step/finish); DirectChatTransport runs an Agent in-process (validateUIMessages -> convertToModelMessages -> agent.stream -> toUIMessageStream), reconnect always null.

### readUIMessageStream + validateUIMessages
readUIMessageStream({message?, stream, onError?, terminateOnError=false}) -> AsyncIterableStream<UIMessage>: runs processUIMessageStream with write() enqueueing structuredClone(state.message) — each element is a progressively more complete snapshot of the same message. validateUIMessages/safeValidateUIMessages: full zod schema mirroring every part variant (575 lines), plus optional metadataSchema, per-data-part dataSchemas, and per-tool input/output validation against tool.inputSchema/outputSchema (input NOT revalidated for output-error state to allow replaying invalid-input failures). safeValidate returns {success,data}|{success,error}.

### Tool definition machinery (packages/provider-utils/src/types/tool.ts, re-exported from 'ai')
Tool<INPUT,OUTPUT,CONTEXT> = FunctionTool (type undefined|'function') | DynamicTool (type 'dynamic') | ProviderDefinedTool (type 'provider', isProviderExecuted:false, id '`{provider}.{name}`', args) | ProviderExecutedTool (isProviderExecuted:true, supportsDeferredResults?). Shared fields: inputSchema: FlexibleSchema (zod or jsonSchema()), outputSchema?, execute?: (input, options)=>OUTPUT|PromiseLike<OUTPUT>|AsyncIterable<OUTPUT> (AsyncIterable yields preliminary results, last value final), contextSchema?, metadata?: JSONObject (propagated to toolMetadata on calls, not sent to the model), title? (deprecated), providerOptions, needsApproval?: boolean|fn (deprecated), description: string | fn({context,experimental_sandbox}), strict?, inputExamples?, and streaming callbacks onInputStart(options), onInputDelta({inputTextDelta, ...options}), onInputAvailable({input, ...options}), toModelOutput({toolCallId,input,output})->ToolResultOutput. ToolExecutionOptions = {toolCallId, messages: ModelMessage[], abortSignal?, context, experimental_sandbox?}. tool() and dynamicTool() are identity functions (dynamicTool adds type:'dynamic') — pure type-level helpers; all invocation logic lives in the generate-text loop.

### MCP client (packages/mcp/src/tool/mcp-client.ts)
createMCPClient({transport: MCPTransportConfig|MCPTransport, onUncaughtError?, maxRetries=0, initialInitializeResult?, clientName='ai-sdk-mcp-client', version='1.0.0', capabilities?}) -> Promise<MCPClient> (constructs DefaultMCPClient then init()). init(): transport.start(); send initialize{protocolVersion: LATEST='2025-11-25', capabilities, clientInfo}; validate result (SUPPORTED_PROTOCOL_VERSIONS = [2025-11-25, 2025-06-18, 2025-03-26, 2024-11-05]); set transport.protocolVersion; send notifications/initialized. Request plumbing: monotonically increasing numeric id; responseHandlers: Map<number, (JSONRPCResponse|Error)=>void>; AbortSignal support with cleanup; per-method capability assertion against serverCapabilities (tools/resources/prompts/completions). Server->client requests handled: 'ping' (reply {}), 'elicitation/create' (via registered handler; else JSON-RPC error -32601). API: listTools/callTool (with retryWithExponentialBackoff when maxRetries>0; retryable = HTTP 408/409/429/5xx or socket codes ECONNRESET/ECONNREFUSED/ETIMEDOUT/EPIPE/ConnectionRefused/ConnectionClosed/FailedToOpenSocket), listResources/readResource/listResourceTemplates, experimental_listPrompts/getPrompt, complete, close. onClose flushes all pending handlers with 'Connection closed' error.

### MCP tools -> SDK tools conversion
tools({schemas:'automatic'|Record<name,{inputSchema,outputSchema?}>}) = listTools() + toolsFromDefinitions(). For each server tool: metadata = {clientName, toolName, title?, app?} (McpProviderMetadata; app from _meta ui extension for MCP Apps). schemas==='automatic' -> dynamicTool({inputSchema: jsonSchema({...server inputSchema, additionalProperties:false}), execute, toModelOutput: mcpToModelOutput}); typed schemas -> tool({inputSchema: schemas[name].inputSchema, outputSchema?}). execute(args, options): throwIfAborted; callTool({name, arguments, signal}); if result.isError return raw result; if outputSchema, extractStructuredContent (validate result.structuredContent, else parse first text content as JSON, else throw MCPClientError). mcpToModelOutput: content array mapped text->{type:'text'}, image->{type:'file',mediaType,data:{type:'data'}}, other->JSON.stringify'd text; no content array -> {type:'json'}. Resulting tools carry `_meta` passthrough. McpToolSet type: Record<string, Tool & required execute>.

### MCP transports (stdio / SSE / Streamable HTTP)
MCPTransport iface: start(), send(JSONRPCMessage), close(), onclose/onerror/onmessage callbacks, protocolVersion/setProtocolVersion. createMcpTransport dispatches config.type 'sse'|'http'. StdioMCPTransport: node spawn(command,args,{env: whitelist-inherited (HOME,LOGNAME,PATH,SHELL,TERM,USER; win32 set differs) merged with custom, stdio:['pipe','pipe',stderr??'inherit'], shell:false, signal, cwd}); messages are JSON.stringify(msg)+'\n' to stdin; stdout buffered in ReadBuffer splitting on '\n', each line parsed+validated. SseMCPTransport (legacy HTTP+SSE): GET url with Accept: text/event-stream; waits for 'endpoint' SSE event giving POST URL (must be same-origin, else fatal); 'message' events parsed as JSON-RPC; send() POSTs JSON to endpoint; 401 triggers OAuth auth() flow once then retry. HttpMCPTransport (Streamable HTTP): send() POSTs each message with Accept: 'application/json, text/event-stream' and headers mcp-protocol-version + mcp-session-id (session id captured from response header, onSessionIdChange/onSessionExpired hooks, 404 with session id => expire session); response 202 => fire-and-forget (start inbound GET SSE if not running); application/json => one-or-array of messages dispatched; text/event-stream => per-request SSE reader loop. Optional inbound GET SSE stream: 405 => server doesn't support it (fine); resumable via last-event-id header tracking event ids; reconnection backoff initial 1000ms, factor 1.5, cap 30000ms, maxRetries 2. close() sends DELETE for session (if terminateSessionOnClose) then aborts. OAuth (oauth.ts, 1332 lines): full RFC 8414/9728 discovery, PKCE authorization, token exchange/refresh, dynamic client registration; auth() returns 'AUTHORIZED'|'REDIRECT'; both HTTP transports auto-retry once on 401. redirect mode defaults to 'error' (reject redirects).

### JSON-RPC message layer (packages/mcp/src/tool/json-rpc-message.ts)
JSONRPCMessage = union of Request{jsonrpc:'2.0', id: string|int, method, params?}, Notification{jsonrpc, method, params?} (no id), Response{jsonrpc, id, result}, Error{jsonrpc, id, error:{code:int, message, data?}} — all zod .strict(). validateJSONRPCMessage / parseJSONRPCMessage(text) helpers. MCP result schemas in types.ts: ListToolsResult{tools:[{name,title?,description?,inputSchema:{type:'object',properties?},outputSchema?,annotations?,_meta?}],nextCursor?}, CallToolResult = {content:[text|image|resource|resource_link], structuredContent?, isError?} | {toolResult}, InitializeResult{protocolVersion,capabilities,serverInfo,instructions?}, plus resources/prompts/completion/elicitation schemas — all 'loose' (extra keys allowed), unlike the strict UI chunk schema.

### Portability verdict per module
PORT (core): all of ui-message-stream/ (chunk types, SSE encode, createUIMessageStream+writer, handleUIMessageStreamFinish, toUIMessageChunk/Stream, readUIMessageStream); all of text-stream/; ui-messages.ts types + type guards; processUIMessageStream reducer; convertToModelMessages; validateUIMessages; AbstractChat + ChatTransport + HttpChatTransport/DefaultChatTransport/TextStreamChatTransport/DirectChatTransport (client via std.http.Client); serial-job-executor, consume-stream, prepare-headers, write-to-server-response (reinterpreted for std.http.Server); entire packages/mcp (client, all 3 transports, JSON-RPC layer; OAuth optionally phased); tool/dynamicTool machinery from provider-utils. SKIP: packages/react hooks (useChat/useCompletion/useObject wrap AbstractChat with useSyncExternalStore + throttle — framework binding), convertFileListToFileUIParts (browser FileList/FileReader DOM API), chat.react.ts. NEUTRAL: use-completion.ts in ai/ui is only option types (UseCompletionOptions/CompletionRequestOptions) consumed by framework packages; callCompletionApi is framework-agnostic (drives /api/completion via setter callbacks) and portable if completion API surface is wanted. Deprecated result methods (toUIMessageStreamResponse etc. on StreamTextResult) delegate to the standalone helpers — port the standalone forms.

## Port implications
Natural Zig mappings: UIMessageChunk and UIMessagePart are tagged unions — ideal Zig tagged unions, except `data-{name}` and `tool-{name}` dynamic type strings, which need a variant carrying a name slice (e.g. .data = {name, id, json}) with the wire type reconstructed at serialization. Since Zig lacks TS generics-over-JSON, METADATA/DATA/tool input/output collapse to std.json.Value plus optional runtime JSON-Schema validation (zod schemas must be reimplemented as hand-written parsers/validators; the strict UI chunk schema and loose MCP schemas differ deliberately — preserve strict vs loose). WHATWG stream pipelines become std.Io Reader/Writer adapters: JsonToSseTransformStream is a trivial writer wrapper; processUIMessageStream is a pull-loop reducer over a chunk iterator mutating an arena-allocated UIMessage. Hard parts: (1) createUIMessageStream's writer.merge() requires spawning concurrent readers feeding one channel and draining a dynamic promise list — maps to io.async tasks plus an MPSC queue and completion counting; (2) structuredClone snapshots (onStepEnd, readUIMessageStream) need explicit deep-copy — arena clone or serialize/reparse; (3) AbortController/AbortSignal maps to std.Io cancellation; every transport and tool execute takes a cancellation token; (4) SerialJobExecutor becomes a mutex or single consumer task; (5) SSE parsing (eventsource-parser) must be hand-written: event/data/id fields, multi-line data, [DONE] sentinel, last-event-id resume; (6) MCP stdio transport maps to std.process.Child with piped stdin/stdout + newline framing; Streamable HTTP needs a long-lived GET SSE task with backoff reconnect concurrent with per-POST response readers — a good stress test of std.Io concurrency. Tool is pure data: struct with JSON schema, optional execute fn pointer + context pointer, onInputStart/Delta/Available hooks, toModelOutput — directly expressible in C ABI as function pointers + user_data, and the MCP-to-tool bridge (dynamicTool with execute=callTool closure) validates that design. For FFI, encode chunks/messages as JSON strings across the boundary; Python/Rust wrappers re-expose typed unions. Exclude React/DOM code; keep AbstractChat — it gives Python/Rust idiomatic client bindings for free.
