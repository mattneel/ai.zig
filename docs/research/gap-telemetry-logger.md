# gap:Telemetry (tracingChannel dispatcher) + logger/warning system internals

## Summary
The AI SDK telemetry layer has two parallel surfaces. (1) A callback-integration surface: `Telemetry` interface (~17 optional callbacks + 2 execute-wrappers) that users register globally via `registerTelemetry()` (pushes onto `globalThis.AI_SDK_TELEMETRY_INTEGRATIONS`) or per-call via `TelemetryOptions.integrations` (local REPLACES global). `createTelemetryDispatcher` fans events out to all integrations (parallel, errors swallowed via Promise.allSettled), stamping `{recordInputs, recordOutputs, functionId}` onto every event (`InferTelemetryEvent`). `isEnabled === false` returns an empty dispatcher `{}` — every call site uses optional chaining so all telemetry work vanishes. (2) A Node diagnostics_channel surface: one channel `'ai:telemetry'` used as a `tracingChannel`, publishing messages `{type: 'generateText'|'streamText'|'step'|'languageModelCall'|'executeTool'|'embed'|'rerank', event}`. Fast path: `tracingChannel.hasSubscribers === false` → direct passthrough; non-Node runtimes (`process.release?.name !== 'node'`) or failed `process.getBuiltinModule('node:diagnostics_channel')` → passthrough. Streaming ops use a manual span protocol (`start.runStores` + `end.publish` at setup, `error.publish`/`asyncEnd.publish` on completion promise, AsyncResource for later context re-entry). Object generation and embedMany get NO channel spans, only callbacks. The `@ai-sdk/otel` package does NOT use the channel — its `OpenTelemetry` class implements the `Telemetry` callback interface, keeping per-callId span state in a Map and emitting gen_ai SemConv spans (`invoke_agent {model}`, `step {n}`, `chat {model}`, `execute_tool {name}`, `embeddings {model}`, `rerank {model}`). Warnings are logged synchronously after every model call via `logWarnings` gated by global `AI_SDK_LOG_WARNINGS` (false=off, function=custom, unset=process.emitWarning/console.warn + one-time info message). Entanglement with core control flow is deep: ~60 call sites across 7 operations, and telemetry callbacks are AWAITED at each lifecycle point.

## Key findings
### Channel vocabulary (tracing-channel.ts)
Single channel name: AI_SDK_TELEMETRY_TRACING_CHANNEL = 'ai:telemetry'. TelemetryTracingEventType = 'generateText' | 'streamText' | 'step' | 'languageModelCall' | 'executeTool' | 'embed' | 'rerank'. Message shape: TelemetryTracingChannelMessage<EVENT> = { readonly type: TelemetryTracingEventType; readonly event: EVENT } — the SAME message object is mutated with `result` (on success) / `error` (on failure) before asyncEnd/error publish. Payloads per type: generateText/streamText → GenerateTextStartEvent (augmented with recordInputs/recordOutputs/functionId); step → {callId, stepNumber}; languageModelCall → LanguageModelCallStartEvent; executeTool → {callId, toolCall, messages, toolContext}; embed → EmbedStartEvent; rerank → RerankStartEvent. Which ops get channel spans: generateText (runInTracingChannelSpan wraps whole op + per-step 'step' + 'languageModelCall' + 'executeTool'), streamText (startTracingChannelContext for op + per-step, runInTracingChannelSpan for tools, executeLanguageModelCall for model calls), embed and rerank (single op span). generateObject/streamObject/embedMany get NO channel spans (callbacks only).

### Node tracingChannel semantics relied upon
From node:diagnostics_channel, tracingChannel(name) returns 5 sub-channels (start/end/asyncStart/asyncEnd/error) plus: `hasSubscribers` boolean (the no-subscriber fast path), `tracePromise(fn, context)` (publishes start, runs fn inside start-channel bound stores, publishes end synchronously, then on promise settle sets context.result and publishes asyncStart+asyncEnd, or sets context.error and publishes error before asyncEnd), `start.bindStore(store, transform)` (subscriber-side AsyncLocalStorage binding used for parentage), `start.runStores(context, fn)` (publisher-side store entry). Test-verified event sequence for generateText/streamText with one tool call: bindStart generateText → bindStart step → bindStart languageModelCall → asyncEnd languageModelCall → bindStart executeTool → asyncEnd executeTool → asyncEnd step → (repeat step) → asyncEnd generateText. Parentage: step parent=streamText/generateText; languageModelCall and executeTool parent=step. embed/rerank: bindStart embed → asyncEnd embed. asyncEnd message carries `result` (e.g. embed result object, rerank result). Abort with subscriber: onAbort fires exactly once, onError does NOT fire.

### tracing-channel-publisher.ts control flow
runWithTracingChannelSpan(message, execute): loads diagnostics_channel via isNodeRuntime() guard + process.getBuiltinModule('node:diagnostics_channel') (loadBuiltinModule returns undefined on any throw; result promise memoized). If channel null or hasSubscribers===false → `return await execute()`. Otherwise wraps execute with bookkeeping (executionResult/executionError/hasExecutionResult flags) and calls tracingChannel.tracePromise(tracedExecute, message); if tracePromise itself throws, the original result/error is re-surfaced and execute is never called twice. openTelemetryChannelSpanContext({message, completion}): for streams — returns TracingChannelContext = { run<T>(execute: () => T): T }. Manually publishes: start.runStores(context, () => { asyncResource = new AsyncResource('ai.telemetry') }) (captures span context in an AsyncResource), end.publish(context) immediately, then on completion promise settle: error.publish (if error, after setting context.error) and asyncEnd.publish once (guarded by asyncEndPublished flag; context.result set from resolution). asyncStart is never published on this manual path. All publishes wrapped in try/catch (subscriber exceptions never affect SDK behavior). Returns undefined (and swallows completion rejection) when non-Node / no subscribers / no async_hooks.

### TelemetryOptions shape (telemetry-options.ts)
TelemetryOptions<RUNTIME_CONTEXT extends Context, TOOLS extends ToolSet> = { isEnabled?: boolean (default: enabled when an integration is registered; false = full opt-out); recordInputs?: boolean (default true); recordOutputs?: boolean (default true); functionId?: string; includeRuntimeContext?: { [key]?: boolean } (allowlist — context props excluded unless explicitly true); includeToolsContext?: { [toolName]?: { [key]?: boolean } }; integrations?: Arrayable<Telemetry> (per-call integrations, take precedence over global) }. Passed as `telemetry` param on generateText/streamText/generateObject/streamObject/embed/embedMany/rerank and on ToolLoopAgentSettings (deprecated alias experimental_telemetry). InferTelemetryEvent<EVENT> = EVENT & Omit<TelemetryOptions, 'integrations'|'isEnabled'|'includeRuntimeContext'> — i.e. events delivered to integrations carry recordInputs/recordOutputs/functionId.

### Telemetry interface (telemetry.ts) — the integration contract
interface Telemetry { onStart?(OperationStartEvent = GenerateTextStartEvent|GenerateObjectStartEvent|EmbedStartEvent|RerankStartEvent — discriminate on operationId); onStepStart?(GenerateTextStepStartEvent); onLanguageModelCallStart?(LanguageModelCallStartEvent); onLanguageModelCallEnd?(LanguageModelCallEndEvent); onToolExecutionStart?(ToolExecutionStartEvent); onToolExecutionEnd?(ToolExecutionEndEvent); onStepEnd?(GenerateTextStepEndEvent = StepResult); onStepFinish? (deprecated alias, still receives step-end via fan-out); onObjectStepStart?/onObjectStepEnd? (deprecated, object ops); onEmbedStart?/onEmbedEnd?(EmbeddingModelCallStartEvent/EndEvent — per doEmbed, keyed by embedCallId for parallel chunks); onRerankStart?/onRerankEnd?; onEnd?(OperationEndEvent union); onAbort?(GenerateTextAbortEvent = {callId, steps, reason?}); onError?(unknown — actually {callId, error} at all call sites); executeLanguageModelCall?<T>(opts: Partial<LanguageModelCallStartEvent> & {callId: string; execute: () => PromiseLike<T>}): PromiseLike<T>; executeTool?<T>(opts: Partial<ToolExecutionStartEvent> & {callId: string; toolCallId: string; execute}): PromiseLike<T> }. All callbacks: Callback<E> = (event: E) => PromiseLike<void> | void. TelemetryDispatcher adds runInTracingChannelSpan?<T>({type, event, execute}) and startTracingChannelContext?({type, event, completion}) → TracingChannelContext|undefined.

### Dispatcher design (create-telemetry-dispatcher.ts)
createTelemetryDispatcher({telemetry}): if telemetry?.isEnabled === false → return {} (all members undefined; call sites use ?. so everything no-ops and executeTool passes through unwrapped; global integrations ignored). integrations = asArray(telemetry.integrations) if provided else getGlobalTelemetryIntegrations() (local fully replaces global). Each callback key: collect integration[key].bind(integration) (this-preservation for class integrations), wrap each so the event is augmented via augmentEvent(event, {recordInputs, recordOutputs, functionId}) — implemented as Object.assign(Object.create(Object.getPrototypeOf(event)), event, meta) so class getters (StepResult) survive; merge with mergeCallbacks (Promise.allSettled — parallel, per-integration errors silently swallowed, sync throws too). onStepEnd fans out to BOTH onStepEnd and deprecated onStepFinish. executeLanguageModelCall composes all integration wrappers innermost-first around execute, then runs the composed fn inside runWithTracingChannelSpan({type:'languageModelCall', event}). executeTool composes wrappers similarly but does NOT open a channel span itself (executeToolCall opens the 'executeTool' span). Registry (telemetry-registry.ts): registerTelemetry(...integrations) pushes onto globalThis.AI_SDK_TELEMETRY_INTEGRATIONS (lazily created array); getGlobalTelemetryIntegrations() returns it or [].

### RestrictedTelemetryDispatcher (context redaction, generate-text/restricted-telemetry-dispatcher.ts)
generateText/streamText wrap the dispatcher with createRestrictedTelemetryDispatcher({telemetry, includeRuntimeContext, includeToolsContext}) which, before dispatching onStart/onStepStart/onStepEnd/onEnd/onAbort/onToolExecutionStart/onToolExecutionEnd, replaces runtimeContext/toolsContext with shallow copies containing only keys explicitly allowlisted (=== true) in includeRuntimeContext/includeToolsContext, and rebuilds each StepResult via new DefaultStepResult({...}) with filtered contexts (restrictStepResult). Default (no config) = empty context objects. This is a privacy allowlist applied only for telemetry consumers; user-facing onStepEnd etc. callbacks get unfiltered events.

## Port implications
Two-layer split maps cleanly to Zig. Layer 1 (must port 1:1): the Telemetry integration interface + dispatcher. Natural mapping is an Allocator-style vtable: `pub const Telemetry = struct { ptr: *anyopaque, vtable: *const VTable }` where VTable holds ~17 OPTIONAL function pointers (`?*const fn(ctx: *anyopaque, event: *const GenerateTextStartEvent) anyerror!void` etc.) — optionality is load-bearing (dispatcher skips unimplemented methods; per-method no-op when nobody implements). Dispatcher = struct owning `[]const Telemetry` + `TelemetryMeta{record_inputs: ?bool, record_outputs: ?bool, function_id: ?[]const u8}`; instead of JS augmentEvent (prototype-preserving object merge), pass meta as a separate parameter to every callback — semantically identical and avoids copying. Contracts to preserve exactly: isEnabled==false → fully empty dispatcher (all null; executeTool passthrough; globals ignored); local integrations REPLACE globals; callbacks run in parallel and are AWAITED, with per-integration errors swallowed (in std.Io: spawn each callback as an async task, join all, discard errors — `notify` is the one primitive to get right); onStepEnd fan-out to deprecated onStepFinish. Global registry = mutex-guarded process-global ArrayList(Telemetry) (globalThis analog); expose `registerTelemetry` through the C ABI as a struct-of-function-pointers registration (the vtable IS the C ABI: `ai_telemetry_t{ void* ctx; cb fields... }`), giving Python/Rust wrappers idiomatic callback objects. executeLanguageModelCall/executeTool wrappers exist solely for ambient-context propagation (otel does `context.with(ctx, execute)`); Zig has no ambient async context, so replace the continuation-composition with paired enter/exit hooks returning an opaque token (`enterModelCall(callId) ?Token` / `exit(Token)`) or explicit parent-span handles threaded through call state — behaviorally equivalent and far simpler than closure chains; document the deviation. Layer 2 (Node diagnostics_channel tracingChannel): platform-specific with NO consumer even in the otel package — do NOT emulate AsyncLocalStorage/AsyncResource. Preserve its observable value as an optional low-level span-event channel: a global `TracingChannel` with subscriber count (`hasSubscribers` fast path → single atomic load; comptime build option to compile the whole layer out) publishing {type: enum{generate_text, stream_text, step, language_model_call, execute_tool, embed, rerank}, event ptr, then result/error on completion} with the exact start/end/asyncEnd ordering from the test snapshots (streaming ops publish start+end at setup and asyncEnd when the completion future resolves — matches the manual openTelemetryChannelSpanContext protocol, which is easier to port than tracePromise). Subscriber panics/errors must never affect generation (safePublish). Entanglement warning: telemetry event structs reference deep core types (StepResult, ContentPart, ModelMessage, usage) — define events alongside those types, not in a leaf module; ~60 call sites across 7 ops mean the dispatcher must be designed before porting generate-text/stream-text, not retrofitted. Lifetime rule for FFI: event payloads are borrowed, valid only for the callback duration (callbacks are awaited, so this is sound); wrappers must copy. RestrictedTelemetryDispatcher's allowlist filtering (shallow-copy contexts keeping only keys set true) needs StringHashMap-based context representation or per-field bools. logWarnings: global tri-state `union(enum){ default, disabled, custom: *const fn(*const WarningsOptions) void }` + std.once for the first-call info message; default sink formats to stderr (no process.emitWarning analog; keep the exact message format strings incl. 'AI SDK Warning (provider / model):' for parity); must be callable from every op after each model call. The gen_ai SemConv span/attribute inventory from packages/otel is the spec for an eventual ai.zig otel exporter package — port it as a separate module implementing the same Telemetry vtable, with the exact span names (`invoke_agent {model}`, `step {n}`, `chat {model}`, `execute_tool {name}`) and the callId-keyed state-map design (works without ambient context).
